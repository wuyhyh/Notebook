# 02-代码阅读方法与注释规范

## 1 文档目标

本文用于帮助实习生掌握基本的代码阅读方法，并建立良好的注释习惯。

学习完本文后，应能够：

1. 有顺序地阅读一个陌生 C 项目。
2. 从目录结构、入口函数、核心数据结构入手理解代码。
3. 使用 `grep`、`find`、`ctags`、IDE 等工具辅助阅读。
4. 区分“有用注释”和“无效注释”。
5. 写出简洁、准确、可维护的注释。
6. 在修改代码前后进行基本自查。

## 2 为什么要学习代码阅读

## 2.1 项目开发不是只写新代码

实际项目中，大多数时间不是从零开始写代码，而是：

1. 阅读已有代码。
2. 理解已有逻辑。
3. 定位问题。
4. 修改已有模块。
5. 添加小功能。
6. 修复 bug。

如果不会读代码，就很难参与真实项目。

## 2.2 代码阅读的目标

读代码不是把每一行都背下来，而是搞清楚：

1. 这个模块做什么。
2. 输入是什么。
3. 输出是什么。
4. 关键数据结构是什么。
5. 主要调用链是什么。
6. 哪些地方容易出错。
7. 修改代码时应该改哪里。

初学阶段要避免陷入细节，先建立整体理解。

## 3 阅读代码的基本顺序

## 3.1 先看目录结构

拿到一个项目后，先看目录：

```bash
ls
tree -L 2
```

常见目录含义：

```text
src/       源代码
include/   头文件
drivers/   驱动代码
arch/      架构相关代码
docs/      文档
scripts/   脚本
tests/     测试代码
build/     构建输出
```

通过目录结构先判断项目大致分层。

## 3.2 再看 README 或文档

优先阅读：

```text
README.md
docs/
BUILD.md
INSTALL.md
```

重点找：

1. 项目用途。
2. 编译方法。
3. 运行方法。
4. 目录说明。
5. 依赖环境。
6. 常见问题。

不要一上来就直接打开最复杂的源文件。

## 3.3 找入口

C 程序常见入口是：

```c
int main(void)
```

可以搜索：

```bash
grep -R "int main" -n .
```

驱动、库、内核模块中不一定有 `main()`，可能入口是：

```c
module_init(xxx_init);
platform_driver_register(&xxx_driver);
xxx_probe();
```

阅读嵌入式或驱动代码时，常见入口包括：

1. `init` 函数。
2. `probe` 函数。
3. `open` 函数。
4. `read` 函数。
5. `write` 函数。
6. `ioctl` 函数。
7. 中断处理函数。
8. 回调函数。

## 3.4 找核心数据结构

C 项目中，结构体通常是理解代码的关键。

搜索：

```bash
grep -R "struct xxx" -n .
```

重点关注：

1. 结构体表示什么对象。
2. 成员变量分别表示什么。
3. 哪些函数会创建它。
4. 哪些函数会修改它。
5. 生命周期从哪里开始，到哪里结束。

示例：

```c
struct uart_port {
    unsigned long base;
    unsigned int irq;
    unsigned int baudrate;
};
```

这个结构体可能就是理解 UART 驱动的核心。

## 3.5 找主要调用链

从入口函数开始，顺着函数调用往下看。

例如：

```text
main()
  -> board_init()
  -> uart_init()
  -> gpio_init()
```

阅读时可以简单画出调用关系：

```text
入口函数
  初始化硬件
  初始化驱动
  启动主循环
```

不要一开始就跳进每个底层函数的全部细节。

## 4 阅读 C 代码时重点看什么

## 4.1 看函数名

好的函数名通常能说明功能：

```c
uart_init();
gpio_set_value();
device_probe();
read_register();
```

看到函数名后，先猜它大概做什么，再进入函数验证。

## 4.2 看函数参数

函数参数能说明它依赖什么输入：

```c
int uart_send(struct uart_port *port, const char *buf, unsigned int len);
```

可以看出：

1. 操作对象是 `port`。
2. 数据来源是 `buf`。
3. 数据长度是 `len`。
4. 返回值是 `int`，可能表示成功或失败。

## 4.3 看返回值

嵌入式和 Linux C 代码中，常见约定是：

1. 返回 `0` 表示成功。
2. 返回负数表示失败。
3. 返回正数表示数量、长度或状态。

示例：

```c
ret = uart_init(port);
if (ret < 0) {
    return ret;
}
```

读代码时要关注错误处理路径，不要只看成功路径。

## 4.4 看全局变量

全局变量会影响代码行为。

搜索：

```bash
grep -R "static .*=" -n .
```

重点关注：

1. 全局状态在哪里定义。
2. 谁会修改它。
3. 是否有并发访问。
4. 是否会影响多个函数。

全局变量太多时，代码通常更难维护。

## 4.5 看宏定义

宏可能隐藏很多细节。

常见宏：

```c
#define BIT(n) (1UL << (n))
#define UART_CTRL_ENABLE BIT(0)
#define UART_BASE 0x28001000UL
```

读代码时要查清楚：

1. 宏的最终值是什么。
2. 宏是否带参数。
3. 宏是否涉及寄存器 bit。
4. 宏是否影响条件编译。

查看宏定义：

```bash
grep -R "#define UART_CTRL_ENABLE" -n .
```

## 4.6 看条件编译

条件编译会导致不同配置下代码不同：

```c
#ifdef CONFIG_DEBUG
    printf("debug\n");
#endif
```

阅读时要注意：

1. 当前配置是否启用。
2. 不同平台是否走不同代码路径。
3. 某些函数是否只在特定宏下编译。

## 5 使用工具辅助阅读

## 5.1 grep 搜索

搜索函数：

```bash
grep -R "uart_init" -n .
```

搜索结构体：

```bash
grep -R "struct uart_port" -n .
```

搜索宏：

```bash
grep -R "UART_CTRL_ENABLE" -n .
```

`grep` 是最基础、最可靠的代码搜索工具。

## 5.2 find 查找文件

查找 C 文件：

```bash
find . -name "*.c"
```

查找头文件：

```bash
find . -name "*.h"
```

查找 Makefile：

```bash
find . -name "Makefile"
```

## 5.3 ctags

`ctags` 可以生成符号索引，方便跳转函数和结构体定义。

生成索引：

```bash
ctags -R .
```

在 Vim 中可以使用：

```text
Ctrl + ]
```

跳转到定义。

返回：

```text
Ctrl + t
```

## 5.4 IDE 跳转

如果使用 CLion、VS Code 等 IDE，应重点掌握：

1. 跳转到定义。
2. 查找引用。
3. 全局搜索。
4. 查看调用层级。
5. 查看类型定义。
6. 查看编译错误。

IDE 很方便，但不要完全依赖 IDE。命令行搜索能力仍然必须掌握。

## 5.5 git 辅助阅读

查看某一行是谁改的：

```bash
git blame path/to/file.c
```

查看文件历史：

```bash
git log -- path/to/file.c
```

查看某次提交内容：

```bash
git show commit_id
```

阅读代码时，如果不理解为什么这么写，可以查历史提交。

## 6 阅读代码的记录方法

## 6.1 记录模块作用

阅读一个模块时，建议记录：

```text
模块名：
主要功能：
入口函数：
核心结构体：
关键调用链：
重要宏定义：
需要确认的问题：
```

示例：

```text
模块名：uart
主要功能：初始化串口并提供发送接口
入口函数：uart_init()
核心结构体：struct uart_port
关键调用链：main -> board_init -> uart_init
重要宏定义：UART_BASE, UART_CTRL_ENABLE
需要确认的问题：中断模式是否启用
```

## 6.2 记录调用链

可以用简单文本记录：

```text
led_set()
  -> gpio_set_value()
    -> writel()
```

这比直接画复杂图更简单。

## 6.3 记录不确定点

阅读代码时，不确定是正常的。

建议明确记录：

```text
问题 1：xxx_config() 中的 flags 第 3 位含义不清楚
问题 2：为什么 init 失败后没有释放 buffer
问题 3：这个宏只在 CONFIG_DEBUG 下生效，当前配置是否打开
```

不要把猜测当作结论。

## 7 注释的基本原则

## 7.1 注释应该解释为什么

好的注释通常解释：

1. 为什么这样做。
2. 有什么硬件限制。
3. 有什么协议要求。
4. 有什么历史原因。
5. 有什么容易踩坑的地方。

不好的注释只是重复代码本身。

## 7.2 不要写废话注释

不推荐：

```c
/* i add 1 */
i++;
```

不推荐：

```c
/* return 0 */
return 0;
```

这些注释没有提供额外信息。

## 7.3 好的注释示例

```c
/*
 * The controller requires a dummy read after enabling the clock,
 * otherwise the first register write may be lost.
 */
readl(base + CTRL_REG);
```

这个注释解释了为什么要做一次看起来多余的读取。

## 7.4 注释要和代码同步

错误的注释比没有注释更危险。

修改代码时，要同时检查相关注释。

如果注释已经不准确，要及时更新或删除。

## 8 函数注释规范

## 8.1 什么时候需要函数注释

以下函数建议写注释：

1. 对外接口函数。
2. 复杂初始化函数。
3. 涉及硬件时序的函数。
4. 涉及锁、并发、中断的函数。
5. 返回值含义不直观的函数。
6. 参数较多的函数。

简单内部函数可以不写长注释，但函数名要清楚。

## 8.2 普通函数注释示例

```c
/*
 * Initialize the UART controller and configure the default baudrate.
 *
 * Return 0 on success, or a negative error code on failure.
 */
int uart_init(struct uart_port *port)
{
    ...
}
```

注释重点：

1. 函数做什么。
2. 返回值表示什么。
3. 必要时说明参数要求。

## 8.3 参数说明示例

```c
/*
 * Send data through the UART port.
 *
 * @port: UART port instance.
 * @buf: buffer containing data to send.
 * @len: number of bytes to send.
 *
 * Return the number of bytes sent, or a negative error code on failure.
 */
int uart_send(struct uart_port *port, const char *buf, unsigned int len)
{
    ...
}
```

这种注释适合对外接口。

## 9 结构体注释规范

## 9.1 什么时候需要结构体注释

以下结构体建议写注释：

1. 对外暴露的结构体。
2. 核心设备对象。
3. 状态机结构体。
4. 寄存器描述结构体。
5. 生命周期较复杂的结构体。

## 9.2 结构体注释示例

```c
/*
 * struct uart_port - UART controller instance
 * @base: register base address
 * @irq: interrupt number
 * @baudrate: configured baudrate
 */
struct uart_port {
    unsigned long base;
    unsigned int irq;
    unsigned int baudrate;
};
```

成员注释应该说明含义，而不是重复变量名。

## 10 宏和寄存器注释

## 10.1 宏注释

寄存器相关宏应尽量清楚：

```c
#define UART_CTRL_ENABLE     BIT(0)
#define UART_CTRL_TX_ENABLE  BIT(1)
#define UART_CTRL_RX_ENABLE  BIT(2)
```

如果 bit 含义容易误解，可以补充注释：

```c
#define UART_STATUS_TX_EMPTY BIT(0) /* TX FIFO is empty */
#define UART_STATUS_RX_READY BIT(1) /* RX FIFO has data */
```

## 10.2 寄存器偏移注释

```c
#define UART_CTRL_REG    0x00 /* Control register */
#define UART_STATUS_REG  0x04 /* Status register */
#define UART_BAUD_REG    0x08 /* Baudrate register */
```

注释不需要太长，但要帮助读者快速理解寄存器用途。

## 10.3 复杂 bit field 注释

```c
#define UART_BAUD_SHIFT  4
#define UART_BAUD_MASK   (0xf << UART_BAUD_SHIFT)
```

如果字段来自手册，建议注释来源：

```c
/*
 * UART_BAUD field occupies bits [7:4], see controller manual section 3.2.
 */
#define UART_BAUD_SHIFT  4
#define UART_BAUD_MASK   (0xf << UART_BAUD_SHIFT)
```

## 11 代码内注释

## 11.1 复杂逻辑前写注释

```c
/*
 * Reset the controller before changing the baudrate.
 * Some hardware revisions may ignore baudrate writes while enabled.
 */
uart_reset(port);
uart_set_baudrate(port, baudrate);
uart_enable(port);
```

注释解释硬件行为，而不是翻译每一行代码。

## 11.2 分段注释

复杂函数可以用短注释分段：

```c
/* Validate input parameters. */
if (port == NULL || buf == NULL) {
    return -1;
}

/* Wait until TX FIFO has enough space. */
while (!uart_tx_ready(port)) {
    ...
}

/* Write data to TX FIFO. */
for (i = 0; i < len; i++) {
    ...
}
```

分段注释可以帮助读者快速浏览函数结构。

## 11.3 TODO 注释

TODO 应该具体：

```c
/* TODO: Add timeout handling for TX FIFO wait. */
```

不推荐：

```c
/* TODO */
```

如果项目有任务编号，可以写：

```c
/* TODO: Add DMA mode support after task #123 is finished. */
```

## 12 不推荐的注释

## 12.1 重复代码

不推荐：

```c
/* set ret to 0 */
ret = 0;
```

## 12.2 过期注释

不推荐：

```c
/* This function only supports 9600 baud. */
uart_set_baudrate(port, 115200);
```

注释和代码矛盾，会误导读者。

## 12.3 情绪化注释

不推荐：

```c
/* stupid hardware */
```

推荐：

```c
/*
 * Hardware requires a delay after reset before accessing registers.
 */
```

注释要专业，不要写情绪。

## 12.4 无意义作者注释

不推荐在每个修改处写：

```c
/* modified by zhangsan */
```

修改历史应由 Git 记录。

代码注释应解释技术原因，不应替代版本控制。

## 13 Linux 内核风格提示

## 13.1 注释风格

Linux 内核中常见多行注释：

```c
/*
 * This is a multi-line comment.
 * Keep it clear and useful.
 */
```

不推荐 C++ 风格注释大量出现：

```c
// comment
```

普通项目可以使用 `//`，但内核风格 C 代码中更常见 `/* ... */`。

## 13.2 函数命名

函数名建议包含模块前缀：

```c
uart_init();
uart_send();
gpio_set_value();
```

不推荐太泛：

```c
init();
send();
handle();
```

模块前缀可以减少命名冲突。

## 13.3 错误路径注释

复杂错误路径可以适当注释：

```c
err_disable_clk:
    clk_disable(port->clk);
err_free:
    free(port);
    return ret;
```

标签名字本身也应表达清楚含义。

## 14 修改代码前的阅读步骤

## 14.1 先确认需求

修改前先写清楚：

1. 要解决什么问题。
2. 影响哪个模块。
3. 输入输出是否变化。
4. 是否影响已有接口。
5. 是否需要更新文档或测试。

不要还没理解问题就直接改代码。

## 14.2 再定位修改点

定位修改点时，建议顺序：

1. 找入口函数。
2. 找核心结构体。
3. 找关键宏。
4. 找相关调用链。
5. 找错误处理路径。
6. 找配置或编译选项。

## 14.3 最后再动手改

修改时应尽量小步提交：

1. 不要一次改太多无关内容。
2. 不要顺手格式化大段无关代码。
3. 不要混合提交文档、重构、功能和 bug 修复。
4. 改完后编译。
5. 能测试就测试。

## 15 修改代码后的自查

## 15.1 编译检查

至少执行：

```bash
make
```

如果是单文件练习：

```bash
gcc -Wall -Wextra -g -O0 main.c -o app
```

确保没有明显编译错误。

## 15.2 查看差异

提交前执行：

```bash
git diff
```

加入暂存区后执行：

```bash
git diff --cached
```

重点检查：

1. 是否有无关修改。
2. 是否有临时调试代码。
3. 是否有多余空行。
4. 是否有错误注释。
5. 是否误删代码。
6. 是否提交了编译产物。

## 15.3 自己解释修改

提交前应能用几句话说明：

1. 我改了哪个模块。
2. 为什么要改。
3. 怎么改的。
4. 有没有影响其他模块。
5. 如何验证过。

如果自己说不清楚，说明修改还没有完全理解。

## 16 常见错误

## 16.1 一上来就读细节

错误方式：

1. 打开一个几千行文件。
2. 从第一行读到最后一行。
3. 试图理解每一个局部变量。
4. 很快迷失。

更好的方式：

1. 先看目录。
2. 再看文档。
3. 找入口。
4. 找结构体。
5. 看调用链。
6. 最后深入细节。

## 16.2 只看成功路径

很多 bug 在错误路径中。

阅读代码时要看：

1. `if (ret < 0)`
2. `goto err`
3. `return -EINVAL`
4. `return NULL`
5. 资源释放路径。

## 16.3 注释写得太多

注释不是越多越好。

如果代码本身很清楚，不需要每一行都注释。

好的代码加上必要注释，比大量废话注释更好。

## 16.4 注释和代码不一致

修改代码后忘记改注释，会让后续维护者误判。

提交前要检查注释是否仍然准确。

## 16.5 不记录疑问

读不懂的地方如果不记录，很快会忘。

建议把疑问写到笔记里，后续逐个确认。

## 17 简单练习

## 17.1 阅读练习

选择一个小 C 项目，完成以下记录：

```text
项目名称：
主要功能：
编译方法：
入口函数：
核心结构体：
主要调用链：
关键宏定义：
当前不理解的问题：
```

要求：

1. 不需要理解每一行代码。
2. 先写出整体结构。
3. 至少记录 3 个函数的调用关系。

## 17.2 注释练习

给下面函数补充合适注释：

```c
int uart_send(struct uart_port *port, const char *buf, unsigned int len)
{
    if (port == NULL || buf == NULL) {
        return -1;
    }

    /* send data */

    return len;
}
```

参考：

```c
/*
 * Send a data buffer through the UART port.
 *
 * @port: UART port instance.
 * @buf: data buffer to send.
 * @len: number of bytes to send.
 *
 * Return the number of bytes sent, or a negative error code on failure.
 */
int uart_send(struct uart_port *port, const char *buf, unsigned int len)
{
```

## 18 小结

代码阅读和注释能力是参与项目开发的基础能力。

需要重点记住：

1. 读代码先看目录和文档，不要直接钻细节。
2. 找入口函数，理解程序从哪里开始。
3. 找核心结构体，理解代码操作的对象。
4. 顺着调用链读代码。
5. 重点关注函数参数、返回值、错误路径和全局状态。
6. 使用 `grep`、`find`、ctags、IDE 和 Git 辅助阅读。
7. 注释应该解释为什么，而不是重复代码。
8. 复杂函数、核心结构体、硬件限制、协议要求需要注释。
9. 注释必须和代码保持一致。
10. 不要写情绪化、过期、无意义的注释。
11. 修改代码前先理解需求和影响范围。
12. 修改代码后要编译、看 diff、自查提交内容。

会读代码，才能正确修改代码；会写注释，才能让别人继续维护你的代码。
