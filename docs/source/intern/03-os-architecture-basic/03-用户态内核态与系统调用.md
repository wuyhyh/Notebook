# 03-用户态内核态与系统调用

## 1 为什么要区分用户态和内核态

操作系统需要保护系统资源，不能让普通应用程序随意访问硬件、内存和关键数据结构。

例如，普通程序不应该随便做这些事情：

```text
直接读写磁盘任意位置
直接操作网卡寄存器
直接修改其他进程的内存
直接关闭中断
直接修改页表
```

如果所有程序都能随便操作这些资源，系统很容易崩溃，也没有安全性可言。

所以 CPU 和操作系统把程序运行状态分成两类：

```text
用户态
内核态
```

简单理解：

```text
用户态：普通应用程序运行的状态
内核态：操作系统内核运行的状态
```

## 2 用户态是什么

用户态是普通应用程序运行的状态。

例如下面这些程序通常运行在用户态：

```text
shell
vim
gcc
python
浏览器
普通业务程序
```

用户态程序的权限较低，不能直接访问硬件，也不能直接执行某些特权指令。

例如，用户态程序通常不能直接：

```text
操作物理内存
访问设备寄存器
修改内核数据结构
直接控制中断
直接切换页表
```

用户态程序如果需要使用系统资源，必须向内核发起请求。

## 3 内核态是什么

内核态是操作系统内核运行的状态。

内核态拥有更高权限，可以执行特权指令，也可以访问硬件资源和内核数据结构。

例如下面这些工作通常在内核态完成：

```text
进程调度
内存管理
文件系统访问
网络协议栈处理
设备驱动运行
中断处理
系统调用处理
```

可以简单理解为：

```text
用户态负责运行普通程序
内核态负责管理系统资源
```

## 4 用户态和内核态的区别

| 对比项 | 用户态 | 内核态 |
|---|---|---|
| 运行对象 | 普通应用程序 | 操作系统内核 |
| 权限 | 低 | 高 |
| 是否能访问硬件 | 不能直接访问 | 可以访问 |
| 是否能执行特权指令 | 不能 | 可以 |
| 出错影响 | 通常只影响当前进程 | 可能导致整个系统崩溃 |
| 典型代码 | shell、vim、gcc、业务程序 | 调度器、文件系统、驱动程序 |

## 5 为什么用户程序不能直接访问硬件

硬件资源是整个系统共享的，必须由内核统一管理。

例如磁盘、网卡、内存、串口等资源，如果所有程序都能直接访问，就会出现严重问题。

例如：

```text
程序 A 正在写磁盘
程序 B 同时直接改写同一块磁盘区域
文件系统数据可能被破坏
```

再例如：

```text
程序 A 随意访问物理内存
可能读到其他进程的数据
也可能修改内核数据结构
```

所以用户程序不能直接操作硬件，而是要通过内核提供的接口访问系统资源。

## 6 系统调用是什么

系统调用是用户态程序请求内核服务的一种机制。

用户态程序不能直接访问内核资源，但可以通过系统调用向内核发出请求。

例如：

```text
读文件
写文件
创建进程
申请内存
创建线程
发送网络数据
打开设备文件
```

这些操作最终都需要进入内核态，由内核完成。

可以简单理解为：

```text
系统调用 = 用户程序进入内核请求服务的入口
```

## 7 常见系统调用示例

Linux 中常见的系统调用包括：

| 系统调用 | 作用 |
|---|---|
| open | 打开文件 |
| read | 读取文件 |
| write | 写入文件 |
| close | 关闭文件 |
| fork | 创建进程 |
| execve | 执行新程序 |
| wait | 等待子进程 |
| mmap | 建立内存映射 |
| ioctl | 设备控制 |
| socket | 创建网络套接字 |

例如 C 程序中调用：

```c
write(1, "hello\n", 6);
```

这个操作会通过 `write` 系统调用进入内核，让内核把数据写到标准输出。

## 8 系统调用的大致流程

以读取文件为例，用户程序执行：

```c
read(fd, buf, size);
```

大致流程如下：

```text
用户程序调用 read()
        |
        v
C 库函数封装系统调用
        |
        v
CPU 从用户态切换到内核态
        |
        v
内核根据系统调用号找到对应处理函数
        |
        v
内核检查参数是否合法
        |
        v
内核执行文件读取操作
        |
        v
结果返回给用户程序
        |
        v
CPU 从内核态返回用户态
```

注意，应用程序一般不是直接手写系统调用指令，而是通过 C 库函数间接发起系统调用。

## 9 C 库函数和系统调用的关系

很多时候，用户程序调用的是 C 库函数，而不是直接调用系统调用。

例如：

```c
printf("hello\n");
```

`printf` 是 C 库函数，不是系统调用。

它内部可能会经过缓冲区处理，最后再调用 `write` 系统调用输出数据。

关系可以这样理解：

```text
应用程序
  |
  v
C 库函数 printf/fread/malloc
  |
  v
系统调用 write/read/brk/mmap
  |
  v
Linux 内核
```

不是所有 C 库函数都会触发系统调用。

例如普通字符串处理函数：

```c
strlen()
strcpy()
memcmp()
```

这些函数通常只在用户态执行，不需要进入内核。

## 10 用户态到内核态的切换

用户程序发起系统调用时，CPU 会从用户态切换到内核态。

这种切换不是普通函数调用，而是一次受控的权限切换。

在不同 CPU 架构上，进入内核的指令不同。

例如：

```text
x86_64：syscall
ARM64：svc
```

ARM64 上常见的系统调用入口形式可以理解为：

```asm
svc #0
```

`svc` 表示 supervisor call，用于从用户态请求内核服务。

## 11 系统调用和普通函数调用的区别

| 对比项 | 普通函数调用 | 系统调用 |
|---|---|---|
| 是否切换权限 | 不切换 | 用户态切换到内核态 |
| 执行位置 | 通常仍在用户态 | 进入内核执行 |
| 开销 | 较小 | 较大 |
| 作用 | 调用程序内部逻辑 | 请求操作系统服务 |
| 典型例子 | strlen、add、sort | read、write、open |

普通函数调用只是程序内部跳转。

系统调用会进入内核，需要保存现场、切换权限、检查参数，然后再返回用户态。

所以系统调用的开销比普通函数调用更大。

## 12 系统调用为什么有开销

系统调用开销主要来自：

```text
用户态和内核态切换
寄存器现场保存与恢复
系统调用参数检查
内核权限检查
可能触发调度、IO、阻塞等待
```

所以程序设计中通常不会频繁进行特别小粒度的系统调用。

例如，频繁执行很多次小数据写入：

```c
write(fd, "a", 1);
write(fd, "b", 1);
write(fd, "c", 1);
```

通常不如一次性写入：

```c
write(fd, "abc", 3);
```

这也是为什么 C 库的 `printf`、`fwrite` 等函数通常会使用缓冲区。

## 13 系统调用和驱动程序的关系

设备驱动运行在内核态，用户程序不能直接调用驱动函数。

用户程序通常通过设备文件和系统调用访问驱动。

例如：

```bash
/dev/ttyS0
/dev/input/event0
/dev/nvme0n1
/dev/gpiochip0
```

用户程序可以执行：

```c
open("/dev/xxx", ...);
read(fd, ...);
write(fd, ...);
ioctl(fd, ...);
close(fd);
```

这些系统调用进入内核后，会转到对应驱动程序的 file operations。

典型关系：

```text
用户程序
  |
  v
open/read/write/ioctl
  |
  v
VFS
  |
  v
设备驱动 file_operations
  |
  v
硬件设备
```

## 14 ioctl 是什么

`ioctl` 是一种常见的设备控制系统调用。

普通的 `read` 和 `write` 适合传输数据，但是很多设备还需要控制命令。

例如：

```text
设置串口波特率
获取网卡状态
控制 GPIO 输入输出
配置显示设备参数
向驱动下发特殊命令
```

这些操作经常通过 `ioctl` 完成。

简单理解：

```text
read/write：偏向数据读写
ioctl：偏向控制命令
```

## 15 如何观察系统调用

Linux 中可以使用 `strace` 观察程序执行了哪些系统调用。

例如：

```bash
strace ls
```

观察 `ls` 命令的系统调用。

也可以跟踪指定系统调用：

```bash
strace -e openat,read,write ls
```

观察一个程序写标准输出：

```bash
strace echo hello
```

可以看到类似：

```text
write(1, "hello\n", 6)
```

其中 `1` 表示标准输出文件描述符。

## 16 简单实验

### 16.1 观察 echo 的系统调用

执行：

```bash
strace echo hello
```

重点观察输出中的：

```text
write(1, "hello\n", 6)
```

这说明 `echo` 最终通过 `write` 系统调用把内容输出到终端。

### 16.2 观察 cat 读取文件

准备一个文件：

```bash
echo "test" > a.txt
```

执行：

```bash
strace -e openat,read,write,close cat a.txt
```

可以观察到：

```text
openat(...)
read(...)
write(...)
close(...)
```

这说明 `cat` 读取文件并输出内容，需要通过系统调用请求内核服务。

### 16.3 观察进程创建

执行：

```bash
strace -f bash -c "echo hello"
```

其中：

```text
-f
```

表示跟踪子进程。

可以观察 shell 创建子进程、执行程序、输出内容的过程。

## 17 常见理解误区

### 17.1 用户态不是不能使用系统资源

用户态程序可以使用系统资源，但不能直接使用。

它必须通过系统调用请求内核帮助完成。

### 17.2 系统调用不是普通函数调用

系统调用会触发用户态到内核态的切换，权限级别发生变化。

普通函数调用不会改变 CPU 权限级别。

### 17.3 printf 不是系统调用

`printf` 是 C 库函数。

它可能最终调用 `write` 系统调用，但 `printf` 本身不是系统调用。

### 17.4 驱动程序不是用户态程序

传统 Linux 设备驱动通常运行在内核态。

用户程序通过 `/dev` 设备文件和系统调用访问驱动，而不是直接调用驱动函数。

## 18 总结

用户态、内核态和系统调用的关系可以这样理解：

```text
用户态：运行普通应用程序，权限低
内核态：运行操作系统内核，权限高
系统调用：用户态程序请求内核服务的入口
```

一句话总结：

```text
普通程序运行在用户态，需要访问系统资源时，通过系统调用进入内核态，由内核代为完成操作。
```

对于 Linux 学习来说，理解这一点非常重要。后面学习进程、文件系统、网络、驱动程序、VFS、设备文件、内存管理时，都会反复遇到用户态、内核态和系统调用这三个概念。
