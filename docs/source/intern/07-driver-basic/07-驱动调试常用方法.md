# 07-驱动调试常用方法

## 1. 文档目标

本文用于整理 Linux 驱动开发入门阶段最常用的调试方法。

对于实习生来说，驱动调试的第一目标不是马上会分析复杂内核问题，而是形成一套稳定的排查顺序：

```text
先确认模块是否加载
再确认设备是否存在
再确认驱动是否注册
再确认设备和驱动是否匹配
再确认 probe 是否进入
再确认资源是否获取成功
最后再看寄存器、中断和用户接口
```

完成本文后，应该能够掌握：

1. 如何查看模块加载状态；
2. 如何查看内核日志；
3. 如何确认设备树节点是否生效；
4. 如何确认 platform 设备和驱动是否存在；
5. 如何确认 `probe()` 是否执行；
6. 如何查看 MMIO 资源；
7. 如何查看中断资源；
8. 如何排查 sysfs 属性节点；
9. 如何整理一份有效的问题报告。

本文只整理常用方法，不深入展开 ftrace、kgdb、crash、perf、eBPF 等高级调试工具。

---

## 2. 驱动调试的基本思路

驱动调试不要一开始就猜硬件坏了，也不要一开始就改大量代码。

推荐按下面顺序排查：

```text
1. 当前系统环境是否正确？
2. 模块是否编译成功？
3. 模块是否加载成功？
4. dmesg 是否有错误？
5. 设备树节点是否存在？
6. compatible 是否匹配？
7. platform_device 是否创建？
8. platform_driver 是否注册？
9. probe 是否执行？
10. probe 执行到哪一步失败？
11. reg / irq / clock / reset 等资源是否获取成功？
12. /sys、/proc、/dev 中对应接口是否出现？
13. 硬件寄存器、中断计数、状态位是否符合预期？
```

简单记：

```text
先看系统；
再看模块；
再看设备；
再看匹配；
再看资源；
最后看硬件行为。
```

## 3. 查看当前系统环境

驱动和内核强相关，调试前要先确认当前系统环境。

常用命令：

```bash
uname -a
uname -r
cat /etc/os-release
cat /proc/cpuinfo
cat /proc/cmdline
```

重点关注：

| 命令 | 关注点 |
|---|---|
| `uname -r` | 当前运行内核版本 |
| `cat /proc/cmdline` | 当前启动参数 |
| `cat /etc/os-release` | 当前系统发行版 |
| `cat /proc/cpuinfo` | CPU 架构和核心信息 |

特别是外部模块开发时，必须确认：

```text
模块是给当前正在运行的内核编译的。
```

如果内核版本不匹配，可能出现：

```text
Invalid module format
```

---

## 4. 查看模块信息

查看当前已加载模块：

```bash
lsmod
```

查找某个模块：

```bash
lsmod | grep my_demo
```

查看模块文件信息：

```bash
modinfo my_demo.ko
```

重点看：

```text
filename
license
description
author
vermagic
alias
depends
```

例如：

```bash
modinfo my_demo.ko | grep vermagic
```

用于确认模块和当前内核是否匹配。

如果驱动使用了设备树匹配，并写了：

```c
MODULE_DEVICE_TABLE(of, my_demo_of_match);
```

则 `modinfo` 中可能看到：

```text
alias: of:N*T*Ctraining,my-demo
```

这说明模块中导出了设备树匹配信息。

---

## 5. 加载和卸载模块

加载模块：

```bash
sudo insmod my_demo.ko
```

卸载模块：

```bash
sudo rmmod my_demo
```

查看模块是否加载：

```bash
lsmod | grep my_demo
```

注意：

```text
insmod 成功只说明模块加载成功；
不代表 probe 一定执行；
不代表设备一定工作。
```

如果 `insmod` 报错，要立刻查看：

```bash
dmesg | tail -n 50
```

常见错误包括：

| 现象 | 可能原因 |
|---|---|
| `Invalid module format` | 模块和内核版本不匹配 |
| `Unknown symbol` | 依赖符号不存在或未导出 |
| `Operation not permitted` | 权限不足或模块加载被限制 |
| `File exists` | 模块已经加载 |
| `No such file or directory` | 文件路径错误或依赖问题 |

## 6. 查看内核日志

驱动调试最常用命令：

```bash
dmesg
```

查看最新日志：

```bash
dmesg | tail -n 50
```

实时观察日志：

```bash
dmesg -w
```

也可以使用：

```bash
journalctl -k
journalctl -k -f
```

如果系统使用 systemd，`journalctl -k` 可以查看内核日志。

建议调试时开两个终端：

```text
终端1：dmesg -w
终端2：insmod / rmmod / cat / echo 等操作
```

这样能实时看到驱动打印。

## 7. 驱动日志打印建议

驱动中常用日志接口：

```c
pr_info("message\n");
pr_err("error\n");
dev_info(dev, "message\n");
dev_err(dev, "error\n");
dev_dbg(dev, "debug\n");
```

在驱动代码中，更推荐使用：

```c
dev_info()
dev_err()
dev_dbg()
```

因为它们会带上设备信息，便于判断是哪一个设备实例打印的日志。

示例：

```c
dev_info(dev, "probe start\n");
dev_err(dev, "failed to get irq: %d\n", irq);
```

`probe()` 中建议打印几个关键点：

```c
dev_info(dev, "probe start\n");
dev_info(dev, "reg mapped\n");
dev_info(dev, "irq = %d\n", irq);
dev_info(dev, "probe done\n");
```

这样可以快速判断 `probe()` 执行到哪一步。

## 8. 查看设备树是否生效

很多驱动问题不是 C 代码问题，而是设备树没有生效。

查看运行时设备树：

```bash
ls /proc/device-tree/
```

查找节点：

```bash
find /proc/device-tree/ -name '*demo*'
```

查看 `compatible`：

```bash
strings /proc/device-tree/路径/compatible
```

查看 `status`：

```bash
strings /proc/device-tree/路径/status
```

查看 `reg`：

```bash
hexdump -C /proc/device-tree/路径/reg
```

查看 `interrupts`：

```bash
hexdump -C /proc/device-tree/路径/interrupts
```

注意：

```text
/proc/device-tree 中看到的是当前运行系统实际使用的设备树。
不是源码目录里的 DTS。
```

如果 DTS 源码已经修改，但 `/proc/device-tree/` 中没有变化，通常说明：

```text
系统启动时没有加载新的 DTB。
```

## 9. 反编译当前DTB

有些系统可以通过 `/sys/firmware/fdt` 获取当前启动使用的 DTB。

反编译：

```bash
dtc -I dtb -O dts -o running.dts /sys/firmware/fdt
```

然后搜索节点：

```bash
grep -n "my_demo" -A 20 running.dts
grep -n "training,my-demo" -A 20 running.dts
```

如果系统没有 `/sys/firmware/fdt`，可以改用实际启动的 DTB 文件，例如：

```bash
dtc -I dtb -O dts -o running.dts /boot/dtb/board.dtb
```

但要注意：

```text
/boot 里的 DTB 文件不一定就是当前启动真正使用的 DTB。
最可靠的还是查看 /proc/device-tree 或 /sys/firmware/fdt。
```

## 10. 检查compatible匹配

设备树中：

```dts
compatible = "training,my-demo";
```

驱动中：

```c
static const struct of_device_id my_demo_of_match[] = {
	{ .compatible = "training,my-demo" },
	{ }
};
```

二者必须完全一致。

检查运行时 compatible：

```bash
strings /proc/device-tree/路径/compatible
```

检查模块 alias：

```bash
modinfo my_demo.ko | grep alias
```

常见错误：

```text
短横线和下划线混用
字符串拼写错误
DTS 修改后没有更新 DTB
节点 status 是 disabled
驱动忘记设置 of_match_table
```

---

## 11. 查看platform设备

查看所有 platform 设备：

```bash
ls /sys/bus/platform/devices/
```

查找自己的设备：

```bash
find /sys/bus/platform/devices/ -name '*demo*'
```

也可以按地址查找：

```bash
find /sys/bus/platform/devices/ -name '*28000000*'
```

如果 `/proc/device-tree/` 中有节点，但 `/sys/bus/platform/devices/` 中没有设备，可能原因包括：

1. 节点位置不对；
2. 父节点没有正确枚举子节点；
3. 父节点 `compatible` 不支持 simple-bus 或对应 bus；
4. `#address-cells` / `#size-cells` 设置不匹配；
5. 节点被禁用；
6. DTB 实际内容和预期不同。

---

## 12. 查看platform驱动

查看所有 platform 驱动：

```bash
ls /sys/bus/platform/drivers/
```

查找自己的驱动：

```bash
find /sys/bus/platform/drivers/ -name '*demo*'
```

如果模块已经加载，但这里没有驱动目录，说明 `platform_driver` 可能没有注册成功，或者模块加载过程中失败。

常见原因：

1. 模块初始化失败；
2. `module_platform_driver()` 没有正确使用；
3. 驱动代码没有编译进模块；
4. `insmod` 实际加载的不是你刚编译的 `.ko`。

---

## 13. 查看设备和驱动是否绑定

进入驱动目录：

```bash
ls -l /sys/bus/platform/drivers/my_demo/
```

如果绑定成功，通常会看到设备符号链接，例如：

```text
28000000.my_demo -> ../../../../devices/platform/28000000.my_demo
```

也可以查看设备目录中的 `driver` 链接：

```bash
ls -l /sys/bus/platform/devices/28000000.my_demo/driver
```

如果设备存在、驱动也存在，但是没有绑定，重点检查：

```text
compatible 是否匹配
设备是否已经被其他驱动绑定
驱动 probe 是否返回错误
```

## 14. 手动bind和unbind

sysfs 提供手动绑定和解绑接口。

进入驱动目录：

```bash
cd /sys/bus/platform/drivers/my_demo/
```

解绑设备：

```bash
echo 28000000.my_demo > unbind
```

重新绑定：

```bash
echo 28000000.my_demo > bind
```

解绑通常会触发：

```text
remove()
```

重新绑定通常会触发：

```text
probe()
```

注意：

```text
不要随意对关键设备执行 unbind，例如根文件系统所在存储控制器、正在使用的网卡、时钟控制器等。
```

这个方法适合对自己写的实验驱动进行调试。

## 15. 查看probe是否执行

判断 `probe()` 是否执行，最简单的方法是在 `probe()` 开头加日志：

```c
dev_info(dev, "probe start\n");
```

加载模块后查看：

```bash
dmesg | tail -n 50
```

如果没有看到 `probe start`，说明 `probe()` 没有进入。

排查方向：

```text
1. 模块是否加载？
2. 设备树节点是否存在？
3. compatible 是否一致？
4. status 是否 okay？
5. platform_device 是否创建？
6. platform_driver 是否注册？
7. 设备是否已经被其他驱动绑定？
```

如果看到 `probe start` 但没有 `probe done`，说明 `probe()` 进入了，但中间某一步失败。

这时要在每一步资源获取后加日志并检查返回值。

---

## 16. 查看probe失败原因

`probe()` 中所有可能失败的函数都要检查返回值。

示例：

```c
priv = devm_kzalloc(dev, sizeof(*priv), GFP_KERNEL);
if (!priv) {
	dev_err(dev, "failed to allocate priv\n");
	return -ENOMEM;
}

priv->base = devm_platform_ioremap_resource(pdev, 0);
if (IS_ERR(priv->base)) {
	dev_err(dev, "failed to map registers: %ld\n", PTR_ERR(priv->base));
	return PTR_ERR(priv->base);
}

irq = platform_get_irq(pdev, 0);
if (irq < 0) {
	dev_err(dev, "failed to get irq: %d\n", irq);
	return irq;
}
```

调试原则：

```text
不要忽略返回值。
不要只打印 failed。
要打印具体错误码。
```

## 17. 查看MMIO资源

查看系统 I/O memory 资源：

```bash
cat /proc/iomem
```

搜索地址：

```bash
grep -i 28000000 /proc/iomem
```

如果驱动成功申请 MMIO 资源，可能能看到对应区域。

查看设备树中的 `reg`：

```bash
hexdump -C /proc/device-tree/路径/reg
```

常见问题：

```text
1. reg 地址写错
2. reg 长度写错
3. 地址 cell 数量不匹配
4. 访问了不存在的地址
5. 访问了其他设备的地址
6. 寄存器偏移写错
```

如果访问 MMIO 后出现 kernel panic 或 external abort，要优先怀疑：

```text
地址错误；
设备未上电；
clock/reset 未打开；
访问了不存在的寄存器窗口；
访问宽度不符合硬件要求。
```

## 18. 查看中断信息

查看中断：

```bash
cat /proc/interrupts
```

搜索自己的设备：

```bash
cat /proc/interrupts | grep my_demo
```

实时观察：

```bash
watch -n 1 'cat /proc/interrupts | grep my_demo'
```

关注：

```text
1. 是否出现设备名
2. 中断号是多少
3. 各 CPU 上计数是否变化
4. 是否异常快速增长
5. 触发类型是否符合预期
```

如果 `/proc/interrupts` 中没有设备名，可能是：

```text
platform_get_irq() 失败；
devm_request_irq() 没有执行；
devm_request_irq() 返回错误；
probe 中途失败。
```

如果出现设备名但计数不增长，可能是：

```text
硬件没有触发中断；
设备侧中断没有使能；
中断控制器配置错误；
interrupts 属性写错；
触发类型不对；
中断状态没有产生。
```

如果计数疯狂增长，可能是：

```text
没有正确清中断；
触发类型配置错误；
硬件一直保持中断有效；
状态寄存器处理错误。
```

## 19. 查看sysfs属性节点

查找设备目录：

```bash
find /sys/bus/platform/devices/ -name '*demo*'
```

进入设备目录：

```bash
cd /sys/bus/platform/devices/设备名/
```

查看属性：

```bash
ls -l
```

读取属性：

```bash
cat enable
cat status
```

写入属性：

```bash
echo 1 > enable
echo 0 > enable
```

如果属性文件没有出现，排查：

```text
1. probe 是否执行？
2. device_create_file() 是否成功？
3. 创建的是哪个 device？
4. 进入的是否是正确设备目录？
5. remove 是否已经执行？
```

如果 `cat` 或 `echo` 失败，排查：

```text
1. 权限是否正确？
2. show/store 是否实现？
3. store 是否返回 count？
4. 输入格式是否合法？
5. dev_get_drvdata() 是否能拿到 priv？
```

## 20. 查看字符设备节点

如果驱动创建了字符设备，查看：

```bash
ls -l /dev/mydev
```

查看主次设备号：

```bash
cat /proc/devices
ls -l /dev/mydev
```

手动创建设备节点：

```bash
sudo mknod /dev/mydev c 主设备号 次设备号
```

读写测试：

```bash
cat /dev/mydev
echo hello > /dev/mydev
```

常见问题：

```text
1. cdev 没有注册成功
2. class/device 没有创建成功
3. /dev 节点没有生成
4. file_operations 没有正确填写
5. copy_to_user/copy_from_user 失败
6. 主次设备号不匹配
```

## 21. 查看设备文件和udev

如果驱动使用：

```c
class_create()
device_create()
```

正常情况下，udev 可能会自动在 `/dev` 下创建设备节点。

查看 udev 事件：

```bash
udevadm monitor
```

查看设备信息：

```bash
udevadm info -a -n /dev/mydev
```

如果 `/sys` 中有设备，但 `/dev` 中没有节点，可能是：

```text
1. 没有调用 device_create()
2. udev 没有运行
3. 权限或规则问题
4. class/device 创建失败
```

---

## 22. 查看内核配置

有些驱动功能依赖内核配置。

查看当前内核配置：

```bash
zcat /proc/config.gz | grep CONFIG_XXX
```

如果没有 `/proc/config.gz`，可以查看：

```bash
ls /boot
grep CONFIG_XXX /boot/config-$(uname -r)
```

例如：

```bash
zcat /proc/config.gz | grep CONFIG_OF
zcat /proc/config.gz | grep CONFIG_MODULES
zcat /proc/config.gz | grep CONFIG_DEBUG_FS
```

常见影响：

```text
CONFIG_MODULES
CONFIG_OF
CONFIG_OF_ADDRESS
CONFIG_OF_IRQ
CONFIG_DEBUG_FS
CONFIG_DYNAMIC_DEBUG
```

如果某些接口编译不过或运行时缺失，要考虑内核配置问题。

---

## 23. 查看内核符号

如果模块加载时报：

```text
Unknown symbol xxx
```

可以查看内核导出符号：

```bash
cat /proc/kallsyms | grep xxx
```

也可以查看模块依赖：

```bash
modinfo my_demo.ko | grep depends
```

常见原因：

```text
1. 调用了未导出的内核函数
2. 依赖模块没有加载
3. 内核配置没有打开
4. GPL-only 符号被非 GPL 模块使用
5. 编译头文件和运行内核不一致
```

如果使用了 GPL-only 符号，模块需要：

```c
MODULE_LICENSE("GPL");
```

## 24. 动态调试dynamic debug

有些驱动中使用：

```c
dev_dbg()
pr_debug()
```

默认可能不打印。

如果内核启用了 dynamic debug，可以通过：

```bash
mount -t debugfs none /sys/kernel/debug
cat /sys/kernel/debug/dynamic_debug/control
```

开启某个文件的调试：

```bash
echo 'file my_demo.c +p' > /sys/kernel/debug/dynamic_debug/control
```

关闭：

```bash
echo 'file my_demo.c -p' > /sys/kernel/debug/dynamic_debug/control
```

查看：

```bash
dmesg -w
```

入门阶段不要求深入掌握，但要知道：

```text
dev_dbg() 默认不一定显示。
dev_info() 更适合入门实验确认流程。
```

## 25. debugfs简单查看

如果系统挂载了 debugfs：

```bash
ls /sys/kernel/debug
```

挂载：

```bash
mount -t debugfs none /sys/kernel/debug
```

有些子系统会把调试信息放在 debugfs 下，例如：

```text
/sys/kernel/debug/
```

debugfs 适合调试，不适合作为稳定用户接口。

对于入门阶段，优先掌握：

```text
dmesg
/sys
/proc
```

debugfs 作为后续补充。

## 26. 查看调用栈和oops

如果驱动导致内核异常，`dmesg` 中可能看到：

```text
Unable to handle kernel NULL pointer dereference
Internal error
Oops
Call trace
Kernel panic
```

看到 oops 时，重点记录：

```text
1. 完整 dmesg
2. 出错函数
3. PC/LR 信息
4. Call trace
5. 加载的模块
6. 触发操作步骤
```

常见原因：

```text
1. 空指针访问
2. 错误指针未判断
3. use-after-free
4. 越界访问
5. MMIO 地址错误
6. 中断中调用了不能调用的函数
7. 并发访问导致数据损坏
```

调试时不要只截最后几行，应该保存完整日志。

## 27. 保存日志

调试时要养成保存日志的习惯。

保存 dmesg：

```bash
dmesg > dmesg.log
```

保存实时日志：

```bash
dmesg -w | tee dmesg-live.log
```

保存系统信息：

```bash
uname -a > env.log
cat /proc/cmdline >> env.log
cat /etc/os-release >> env.log
```

保存中断信息：

```bash
cat /proc/interrupts > interrupts.log
```

保存设备树反编译结果：

```bash
dtc -I dtb -O dts -o running.dts /sys/firmware/fdt
```

问题报告中应包含日志，而不是只写一句：

```text
驱动不工作。
```

## 28. 常见问题排查模板

遇到 `probe()` 不进时：

```text
1. lsmod 是否看到模块？
2. dmesg 是否有模块加载错误？
3. /proc/device-tree 中是否有节点？
4. compatible 是否和驱动完全一致？
5. status 是否是 okay？
6. /sys/bus/platform/devices 中是否有设备？
7. /sys/bus/platform/drivers 中是否有驱动？
8. 设备是否已经绑定到其他驱动？
9. 实际启动的 DTB 是否是修改后的 DTB？
```

遇到 `ioremap` 失败时：

```text
1. DTS 中是否有 reg？
2. reg cell 数量是否正确？
3. 地址和长度是否正确？
4. 资源是否已经被其他设备占用？
5. devm_platform_ioremap_resource() 返回值是多少？
6. /proc/iomem 中是否有对应资源？
```

遇到中断不触发时：

```text
1. DTS 中是否有 interrupts？
2. platform_get_irq() 是否成功？
3. devm_request_irq() 是否成功？
4. /proc/interrupts 是否有设备名？
5. 硬件中断是否使能？
6. 中断状态寄存器是否置位？
7. 触发类型是否正确？
8. 是否正确清中断？
```

遇到 sysfs 属性不存在时：

```text
1. probe 是否进入？
2. device_create_file() 是否成功？
3. 设备目录是否正确？
4. 属性名是否正确？
5. remove 是否已经执行？
```

## 31. 调试时的注意事项

驱动调试可能影响整个系统稳定性，所以要注意：

```text
1. 不要随意访问未知 MMIO 地址；
2. 不要随意写硬件控制寄存器；
3. 不要对关键设备随意 unbind；
4. 不要在中断处理函数中大量打印；
5. 不要忽略错误返回值；
6. 不要在生产系统上做危险实验；
7. 修改 DTB 前保留可启动的旧版本；
8. 每次只修改一个关键点，方便定位问题；
9. 记录操作步骤和日志；
10. 出现 kernel panic 后保存完整日志。
```

对于项目开发板，建议保留一个可恢复方案：

```text
1. 可进入 U-Boot；
2. 保留旧内核和旧 DTB；
3. 保留可用 rootfs；
4. 可以通过串口查看日志；
5. 可以通过另一个系统分区修复文件。
```

## 32. 小结

Linux 驱动调试的核心不是乱试，而是按层次排查。

最常用的判断链路是：

```text
模块是否加载
  ↓
设备树是否生效
  ↓
platform_device 是否创建
  ↓
platform_driver 是否注册
  ↓
compatible 是否匹配
  ↓
probe 是否进入
  ↓
资源是否获取成功
  ↓
硬件是否产生预期行为
```

入门阶段必须熟练掌握这些工具：

```text
dmesg
lsmod
modinfo
/proc/device-tree
/sys/bus/platform
/proc/iomem
/proc/interrupts
```

只要能把这些信息查清楚，大多数入门阶段的驱动问题都能定位到明确方向。

不要只说“驱动不工作”，要能说明：

```text
卡在了哪一层。
```
