# 03-虚拟地址物理地址与MMU基础

## 1 为什么要理解地址

在学习 Linux、驱动、内存管理和 PCIe 时，经常会遇到这些概念：

```text
虚拟地址
物理地址
页表
MMU
ioremap
mmap
DMA 地址
```

其中最基础的是虚拟地址、物理地址和 MMU。

简单说：

```text
虚拟地址：程序看到的地址
物理地址：真实硬件内存上的地址
MMU：负责把虚拟地址转换成物理地址的硬件单元
```

理解这三个概念，是继续学习 Linux 内存管理和驱动开发的基础。

## 2 物理地址是什么

物理地址是硬件层面真实存在的地址。

例如一块 DDR 内存从某个地址开始映射到系统地址空间中，CPU 最终访问内存时，需要落到具体的物理地址上。

可以简单理解为：

```text
物理地址 = 硬件总线上的真实地址
```

例如系统中可能存在这样的物理地址范围：

```text
0x80000000 ~ 0xffffffff
```

这表示某段 DDR 内存被映射到了这个物理地址范围。

注意，物理地址不一定只对应 DDR，也可能对应设备寄存器区域。

例如：

```text
DDR 内存区域
UART 控制器寄存器区域
GPIO 控制器寄存器区域
PCIe ECAM 配置空间
PCIe BAR 空间
```

这些都可能占用系统的物理地址空间。

## 3 虚拟地址是什么

虚拟地址是程序运行时看到的地址。

用户程序里面打印一个指针，例如：

```c
printf("%p\n", ptr);
```

看到的通常是虚拟地址，而不是物理地址。

例如：

```text
0x7ffcc2a01000
```

这个地址是进程虚拟地址空间中的地址。

可以简单理解为：

```text
虚拟地址 = 软件看到的地址
```

每个进程通常都有自己独立的虚拟地址空间。两个不同进程中即使出现相同的虚拟地址，也不代表它们访问的是同一块物理内存。

## 4 为什么需要虚拟地址

如果所有程序都直接使用物理地址，会有很多问题。

例如：

```text
程序 A 可能误写程序 B 的内存
程序可能误写内核内存
程序装载地址难以统一
内存碎片管理困难
进程之间缺乏隔离
```

引入虚拟地址后，操作系统可以给每个进程提供一个看起来独立、连续的地址空间。

优点包括：

```text
进程隔离更好
程序编写更简单
内存管理更灵活
可以实现按需分配
可以实现换页和内存保护
```

所以现代操作系统通常都会使用虚拟内存机制。

## 5 MMU 是什么

MMU 是 Memory Management Unit，中文一般叫内存管理单元。

它是 CPU 中负责地址转换和权限检查的硬件单元。

当程序访问一个虚拟地址时，CPU 不会直接把这个地址送到内存总线上，而是先经过 MMU 转换。

基本流程：

```text
CPU 发出虚拟地址
        |
        v
MMU 查询页表
        |
        v
转换成物理地址
        |
        v
访问 DDR 或设备寄存器
```

可以简单理解为：

```text
MMU = 虚拟地址到物理地址的翻译器
```

## 6 页表是什么

页表是操作系统建立的一张地址映射表。

它记录了虚拟地址和物理地址之间的对应关系。

例如可以粗略理解为：

```text
虚拟地址 0x40000000 -> 物理地址 0x90000000
虚拟地址 0x40001000 -> 物理地址 0x90001000
虚拟地址 0x40002000 -> 物理地址 0x90002000
```

实际系统中，页表通常不是一张简单的大表，而是多级页表。

例如 ARM64 和 x86_64 都使用多级页表结构。

页表中不仅记录地址映射，还记录访问权限，例如：

```text
是否可读
是否可写
是否可执行
用户态是否可访问
是否允许缓存
```

## 7 页是什么

虚拟内存管理通常不是一个字节一个字节地映射，而是按页管理。

常见页大小是：

```text
4KB
```

也就是：

```text
1 page = 4096 bytes
```

如果页大小是 4KB，那么虚拟地址到物理地址的映射通常以 4KB 为基本单位。

例如：

```text
虚拟页 0x40000 -> 物理页 0x90000
```

页内偏移保持不变。

简单说：

```text
虚拟地址 = 虚拟页号 + 页内偏移
物理地址 = 物理页号 + 页内偏移
```

## 8 地址转换示意

假设页大小是 4KB。

某个虚拟地址：

```text
0x40001234
```

可以拆成：

```text
虚拟页号：0x40001
页内偏移：0x234
```

如果页表中记录：

```text
虚拟页 0x40001 -> 物理页 0x90001
```

那么转换后的物理地址就是：

```text
0x90001234
```

也就是页号被转换，页内偏移保持不变。

## 9 TLB 是什么

如果每次访问内存都去查页表，速度会很慢。

所以 CPU 内部通常有 TLB。

TLB 是 Translation Lookaside Buffer，可以理解为地址转换缓存。

它缓存最近使用过的虚拟地址到物理地址的转换结果。

简单理解：

```text
页表：完整的地址映射表
TLB：最近用过的地址映射缓存
```

访问流程可以粗略理解为：

```text
CPU 访问虚拟地址
        |
        v
先查 TLB
        |
        +-- 命中：直接得到物理地址
        |
        +-- 未命中：查页表，再更新 TLB
```

TLB 对系统性能很重要。

## 10 用户态地址和内核态地址

Linux 中，用户程序使用的是用户态虚拟地址。

内核也有自己的虚拟地址空间，叫内核虚拟地址空间。

可以简单分成：

```text
用户态虚拟地址
内核态虚拟地址
物理地址
```

用户程序不能直接访问内核虚拟地址。

内核可以访问自己的内核地址空间，也可以在特定情况下访问用户态地址，但需要使用专门接口。

例如驱动中不能随便直接解引用用户传进来的指针，而应该使用：

```c
copy_from_user()
copy_to_user()
```

这是为了避免非法地址访问和安全问题。

## 11 驱动开发中为什么经常遇到物理地址

驱动程序经常需要操作硬件。

硬件寄存器通常位于某个物理地址范围。

例如设备树中可能有：

```dts
uart0: serial@28001000 {
    reg = <0x0 0x28001000 0x0 0x1000>;
};
```

这里的 `0x28001000` 通常表示设备寄存器的物理地址。

但是内核代码不能直接把这个物理地址当成普通指针访问。

通常需要先映射：

```c
void __iomem *base;

base = ioremap(0x28001000, 0x1000);
```

之后再通过专门接口读写：

```c
readl(base + offset);
writel(value, base + offset);
```

## 12 ioremap 是什么

`ioremap()` 用于把设备物理地址映射到内核虚拟地址。

原因是：内核代码访问地址时，使用的仍然是虚拟地址。

所以驱动访问设备寄存器的大致过程是：

```text
设备寄存器物理地址
        |
        v
ioremap()
        |
        v
内核虚拟地址
        |
        v
readl()/writel() 访问设备
```

注意，设备寄存器不是普通内存，访问时要使用 `readl()`、`writel()` 这类接口，而不是直接用普通指针读写。

## 13 mmap 是什么

`mmap()` 是用户态常见的系统调用，可以把文件或设备的一段内容映射到用户进程虚拟地址空间。

例如用户程序可以通过 `mmap()` 把设备缓冲区映射到自己的地址空间中。

大致关系：

```text
用户程序虚拟地址
        |
        v
mmap()
        |
        v
映射到文件、设备缓冲区或物理页
```

在驱动开发中，`mmap()` 常用于让用户程序直接访问驱动分配的缓冲区，从而减少数据拷贝。

## 14 DMA 地址简单理解

DMA 是设备直接访问内存的机制。

CPU 访问内存时使用虚拟地址，然后经过 MMU 转换成物理地址。

但是设备做 DMA 时，设备看到的地址不一定等于 CPU 的虚拟地址。

在简单系统中，可以粗略理解为设备使用物理地址。

但在现代系统中，可能还存在 IOMMU，这时设备使用的是 I/O 虚拟地址。

所以驱动中不能随便把普通指针地址直接交给硬件做 DMA，而应该使用 DMA API。

例如：

```c
dma_addr_t dma_handle;

buf = dma_alloc_coherent(dev, size, &dma_handle, GFP_KERNEL);
```

其中：

```text
buf：CPU 使用的内核虚拟地址
dma_handle：设备 DMA 使用的地址
```

这是驱动开发中非常重要的一点。

## 15 常见命令和观察方法

查看内存总体情况：

```bash
free -h
```

查看进程虚拟内存映射：

```bash
cat /proc/<PID>/maps
```

例如查看当前 shell：

```bash
cat /proc/$$/maps
```

查看内核打印中的内存信息：

```bash
dmesg | grep -i memory
```

查看设备树中的内存节点：

```bash
ls /proc/device-tree/memory*
```

查看设备树中的 reg 属性可以使用：

```bash
hexdump -C /proc/device-tree/memory@*/reg
```

也可以使用 `dtc` 反编译设备树后查看：

```bash
dtc -I fs -O dts /proc/device-tree > running.dts
```

## 16 常见理解误区

### 16.1 虚拟地址不是假地址

虚拟地址不是没有意义的地址。

它是程序实际使用的地址，只是需要经过 MMU 转换后才能访问真实硬件资源。

### 16.2 物理地址不一定都是 DDR

物理地址空间中不仅可以有内存，也可以有设备寄存器、PCIe 空间等区域。

所以看到一个物理地址时，要结合平台地址映射表、设备树或 ACPI 信息判断它对应什么硬件资源。

### 16.3 用户程序看到的指针不是物理地址

用户程序中的普通指针通常是虚拟地址。

不能把用户态指针直接当成物理地址交给硬件使用。

### 16.4 驱动中也不是直接使用物理地址

内核驱动访问设备寄存器时，通常要先 `ioremap()`，然后使用映射后的内核虚拟地址访问。

### 16.5 DMA 地址不能随便用指针代替

DMA 地址要通过 DMA API 获取。

普通 CPU 指针和设备 DMA 地址不是同一个概念。

## 17 总结

虚拟地址、物理地址和 MMU 的关系可以这样理解：

```text
程序使用虚拟地址
MMU 根据页表进行地址转换
最终访问物理地址对应的内存或设备
```

一句话总结：

```text
虚拟地址是软件看到的地址，物理地址是硬件访问的地址，MMU 负责把两者联系起来。
```

对于 Linux 驱动开发来说，还要特别记住：

```text
用户态指针不能直接当物理地址
设备寄存器物理地址需要 ioremap 后访问
DMA 地址要通过 DMA API 获取
```

这些概念后面学习内存管理、设备树、PCIe BAR、DMA、IOMMU 和驱动开发时都会反复遇到。
