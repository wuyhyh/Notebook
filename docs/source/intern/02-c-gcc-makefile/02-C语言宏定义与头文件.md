# 02-C语言宏定义与头文件

## 1 文档目标

本文用于帮助实习生掌握 C 语言项目中常见的宏定义和头文件使用方法。

学习完本文后，应能够：

1. 看懂常见的 `#define`、`#include`、头文件保护写法。
2. 区分声明和定义。
3. 知道 `.h` 文件和 `.c` 文件分别应该放什么内容。
4. 避免宏定义中的常见错误。
5. 能够阅读简单的驱动、BSP、嵌入式 C 项目代码。

## 2 宏定义基础

## 2.1 什么是宏

宏是 C 预处理器在编译前进行的文本替换。

常见写法：

```c
#define LED_ON      1
#define LED_OFF     0
#define BUFFER_SIZE 1024
```

使用示例：

```c
char buffer[BUFFER_SIZE];
```

预处理后相当于：

```c
char buffer[1024];
```

注意：宏不是变量，它没有类型检查，本质上是文本替换。

## 2.2 宏常量

宏常量常用于定义固定值：

```c
#define UART_BASE_ADDR  0x28001000
#define UART_REG_SIZE   0x1000
#define MAX_RETRY       3
```

相比直接在代码里写数字，宏常量的好处是：

1. 名字表达含义。
2. 统一修改方便。
3. 避免代码中出现大量魔法数字。

不推荐：

```c
if (retry > 3) {
    return -1;
}
```

推荐：

```c
#define MAX_RETRY 3

if (retry > MAX_RETRY) {
    return -1;
}
```

## 2.3 带参数的宏

宏也可以带参数：

```c
#define BIT(n) (1UL << (n))
```

使用示例：

```c
reg |= BIT(3);
```

相当于：

```c
reg |= (1UL << 3);
```

这类宏在寄存器操作中非常常见。

## 2.4 宏参数要加括号

错误写法：

```c
#define SQUARE(x) x * x
```

使用：

```c
int a = SQUARE(1 + 2);
```

展开后：

```c
int a = 1 + 2 * 1 + 2;
```

结果不是 9，而是 5。

正确写法：

```c
#define SQUARE(x) ((x) * (x))
```

宏参数和整体表达式都应加括号，避免优先级问题。

## 2.5 宏不要重复计算参数

下面这个宏有隐患：

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

如果这样使用：

```c
int x = MAX(i++, j++);
```

`i++` 或 `j++` 可能被执行多次，导致结果难以预期。

所以带副作用的表达式不要直接传给宏：

```c
int a = i++;
int b = j++;
int x = MAX(a, b);
```

在复杂场景下，优先考虑使用 `static inline` 函数。

## 3 常见宏写法

## 3.1 BIT 宏

寄存器操作中经常需要表示某一位：

```c
#define BIT(n) (1UL << (n))
```

使用示例：

```c
#define UART_CTRL_ENABLE BIT(0)
#define UART_CTRL_TX_EN  BIT(1)
#define UART_CTRL_RX_EN  BIT(2)
```

设置 bit：

```c
reg |= UART_CTRL_ENABLE;
```

清除 bit：

```c
reg &= ~UART_CTRL_ENABLE;
```

判断 bit：

```c
if (reg & UART_CTRL_ENABLE) {
    /* enabled */
}
```

## 3.2 掩码宏

如果一个字段占多个 bit，可以定义 mask：

```c
#define GENMASK(h, l)     (((~0UL) >> (sizeof(unsigned long) * 8 - 1 - (h))) &      (~0UL << (l)))
```

在实际 Linux 内核代码中，常见写法是直接使用已有的 `GENMASK()`。

示例：

```c
#define UART_BAUD_MASK GENMASK(7, 4)
```

表示 bit 7 到 bit 4 这个字段。

如果暂时不用复杂宏，也可以直接写：

```c
#define UART_BAUD_MASK (0xf << 4)
```

## 3.3 字段偏移宏

多 bit 字段通常需要 mask 和 shift 配合使用：

```c
#define UART_BAUD_SHIFT 4
#define UART_BAUD_MASK  (0xf << UART_BAUD_SHIFT)
```

写入字段：

```c
reg &= ~UART_BAUD_MASK;
reg |= (value << UART_BAUD_SHIFT) & UART_BAUD_MASK;
```

读取字段：

```c
value = (reg & UART_BAUD_MASK) >> UART_BAUD_SHIFT;
```

## 3.4 do while 0 宏

如果宏内部有多条语句，推荐使用 `do { } while (0)` 包起来：

```c
#define CLEAR_AND_SET(reg, clear, set)     do {         (reg) &= ~(clear);         (reg) |= (set);     } while (0)
```

这样可以让宏在 `if/else` 中表现得像一条普通语句。

不推荐：

```c
#define CLEAR_AND_SET(reg, clear, set)     (reg) &= ~(clear);     (reg) |= (set)
```

这种写法在条件语句中容易出错。

## 3.5 字符串宏

宏可以把参数转换成字符串：

```c
#define TO_STRING(x) #x
```

示例：

```c
const char *s = TO_STRING(hello);
```

结果相当于：

```c
const char *s = "hello";
```

字符串宏在调试、日志、版本号生成中可能会遇到。

## 4 头文件基础

## 4.1 什么是头文件

头文件通常以 `.h` 结尾，用来放公共声明、宏定义、结构体定义等内容。

例如：

```c
/* uart.h */
#ifndef UART_H
#define UART_H

#define UART_BASE_ADDR 0x28001000

int uart_init(void);
void uart_putc(char c);

#endif
```

对应的 `.c` 文件中实现函数：

```c
/* uart.c */
#include "uart.h"

int uart_init(void)
{
    return 0;
}

void uart_putc(char c)
{
    /* send char */
}
```

## 4.2 include 的两种写法

包含系统头文件：

```c
#include <stdio.h>
#include <stdint.h>
```

包含项目自己的头文件：

```c
#include "uart.h"
#include "gpio.h"
```

一般规则：

1. `<...>` 用于系统头文件或编译器搜索路径中的头文件。
2. `"..."` 用于当前项目中的头文件。

## 4.3 头文件保护

头文件需要防止重复包含。

常见写法：

```c
#ifndef UART_H
#define UART_H

/* declarations */

#endif
```

也可以写得更具体，避免名字冲突：

```c
#ifndef __DRIVERS_UART_H__
#define __DRIVERS_UART_H__

/* declarations */

#endif
```

头文件保护的作用是：同一个头文件被多次 `#include` 时，只展开一次。

## 4.4 pragma once

有些项目也会使用：

```c
#pragma once
```

它也能避免头文件重复包含。

但是在内核、BSP、可移植 C 项目中，传统的 `#ifndef/#define/#endif` 更常见，也更稳妥。

## 5 声明和定义

## 5.1 函数声明

头文件中通常放函数声明：

```c
int uart_init(void);
void uart_putc(char c);
```

函数实现放在 `.c` 文件中：

```c
int uart_init(void)
{
    return 0;
}
```

声明告诉编译器：这个函数存在。

定义真正生成函数代码。

## 5.2 变量声明

如果一个全局变量需要被多个 `.c` 文件使用，可以在头文件中写 `extern` 声明：

```c
/* board.h */
#ifndef BOARD_H
#define BOARD_H

extern int board_id;

#endif
```

然后在某一个 `.c` 文件中定义它：

```c
/* board.c */
#include "board.h"

int board_id = 0;
```

不要在头文件中直接定义普通全局变量：

```c
/* 不推荐 */
int board_id = 0;
```

因为多个 `.c` 文件包含这个头文件后，可能导致重复定义。

## 5.3 结构体定义

公共结构体可以放在头文件中：

```c
struct uart_port {
    unsigned long base;
    unsigned int irq;
    unsigned int baudrate;
};
```

如果结构体只在某个 `.c` 文件内部使用，就不要放到头文件中。

## 5.4 枚举定义

枚举也经常放在头文件中：

```c
enum uart_parity {
    UART_PARITY_NONE,
    UART_PARITY_ODD,
    UART_PARITY_EVEN,
};
```

相比一组无关联的宏，枚举更适合表示一类有限选项。

## 6 头文件应该放什么

## 6.1 适合放在头文件中的内容

头文件中适合放：

1. 宏常量。
2. 结构体定义。
3. 枚举定义。
4. 函数声明。
5. `extern` 变量声明。
6. `static inline` 小函数。

示例：

```c
#ifndef GPIO_H
#define GPIO_H

#define GPIO_INPUT  0
#define GPIO_OUTPUT 1

struct gpio_desc {
    unsigned int number;
    unsigned int direction;
};

int gpio_init(unsigned int number);
int gpio_set_value(unsigned int number, int value);

#endif
```

## 6.2 不适合放在头文件中的内容

头文件中一般不要放：

1. 普通函数定义。
2. 普通全局变量定义。
3. 很大的实现代码。
4. 只被一个 `.c` 文件使用的内部细节。

不推荐：

```c
/* bad.h */
int counter = 0;

void foo(void)
{
    counter++;
}
```

这种写法容易导致重复定义，也会让代码结构混乱。

## 7 static inline 函数

## 7.1 为什么使用 static inline

有些简单操作可以写成 `static inline` 函数，替代复杂宏。

示例：

```c
static inline unsigned int reg_get_field(unsigned int reg,
                                         unsigned int mask,
                                         unsigned int shift)
{
    return (reg & mask) >> shift;
}
```

相比宏，`static inline` 的优点是：

1. 有类型检查。
2. 不容易重复计算参数。
3. 调试时更清晰。
4. 可读性更好。

## 7.2 适合使用 static inline 的场景

适合写成 `static inline` 的内容：

1. 简单寄存器字段操作。
2. 简单类型转换。
3. 简单包装函数。
4. 对性能敏感但逻辑很短的函数。

示例：

```c
static inline void uart_enable(unsigned int *reg)
{
    *reg |= BIT(0);
}
```

如果函数逻辑较复杂，应放在 `.c` 文件中实现。

## 8 嵌入式项目中的常见头文件组织

## 8.1 示例目录结构

一个简单驱动模块可以这样组织：

```text
drivers/
└── uart/
    ├── uart.c
    ├── uart.h
    └── uart_regs.h
```

其中：

1. `uart.c` 放驱动主要实现。
2. `uart.h` 放对外接口。
3. `uart_regs.h` 放寄存器地址、bit、mask 等定义。

## 8.2 寄存器头文件示例

```c
#ifndef UART_REGS_H
#define UART_REGS_H

#define UART_CTRL_OFFSET     0x00
#define UART_STATUS_OFFSET   0x04
#define UART_BAUD_OFFSET     0x08

#define UART_CTRL_ENABLE     BIT(0)
#define UART_CTRL_TX_ENABLE  BIT(1)
#define UART_CTRL_RX_ENABLE  BIT(2)

#define UART_STATUS_TX_EMPTY BIT(0)
#define UART_STATUS_RX_READY BIT(1)

#endif
```

这类头文件的目标是让寄存器含义清晰，而不是让代码里到处出现裸数字。

## 8.3 对外接口头文件示例

```c
#ifndef UART_H
#define UART_H

struct uart_port {
    unsigned long base;
    unsigned int irq;
};

int uart_init(struct uart_port *port);
int uart_send(struct uart_port *port, const char *buf, unsigned int len);

#endif
```

这个头文件给其他模块使用，不一定需要暴露全部寄存器细节。

## 9 常见错误

## 9.1 宏没有加括号

错误：

```c
#define BIT(n) 1 << n
```

推荐：

```c
#define BIT(n) (1UL << (n))
```

## 9.2 在头文件中定义全局变量

错误：

```c
int debug_enable = 1;
```

推荐：

```c
extern int debug_enable;
```

然后在一个 `.c` 文件中定义：

```c
int debug_enable = 1;
```

## 9.3 头文件互相包含过多

不推荐让多个头文件复杂互相包含。

如果只需要使用结构体指针，可以使用前向声明：

```c
struct device;

int driver_probe(struct device *dev);
```

这样可以减少头文件依赖。

## 9.4 宏名字太短

不推荐：

```c
#define EN 1
#define DIS 0
```

推荐：

```c
#define UART_ENABLE  1
#define UART_DISABLE 0
```

宏名应该带有模块前缀，减少冲突。

## 9.5 把所有内容都放进一个大头文件

不推荐一个项目只有一个巨大的 `common.h`。

更好的方式是按模块拆分：

```text
uart.h
gpio.h
spi.h
board.h
```

这样依赖关系更清楚，也更方便维护。

## 10 建议掌握的习惯

## 10.1 宏定义习惯

建议：

1. 宏名使用大写。
2. 多词之间使用下划线。
3. 宏参数加括号。
4. 整个宏表达式加括号。
5. 多语句宏使用 `do { } while (0)`。
6. 复杂逻辑优先考虑 `static inline`。

## 10.2 头文件习惯

建议：

1. 每个头文件都加头文件保护。
2. `.h` 文件放公共接口，不放实现细节。
3. `.c` 文件放具体实现。
4. 能不暴露的内容就不要暴露。
5. 不要在头文件中定义普通全局变量。
6. 尽量减少头文件之间的复杂依赖。

## 11 简单练习

## 11.1 宏练习

写出以下宏：

1. `BIT(n)`：表示第 n 位。
2. `UART_CTRL_ENABLE`：表示 bit 0。
3. `UART_CTRL_TX_ENABLE`：表示 bit 1。
4. `UART_CTRL_RX_ENABLE`：表示 bit 2。
5. `UART_BAUD_MASK`：表示 bit 7 到 bit 4。

参考：

```c
#define BIT(n)              (1UL << (n))
#define UART_CTRL_ENABLE    BIT(0)
#define UART_CTRL_TX_ENABLE BIT(1)
#define UART_CTRL_RX_ENABLE BIT(2)
#define UART_BAUD_SHIFT     4
#define UART_BAUD_MASK      (0xf << UART_BAUD_SHIFT)
```

## 11.2 头文件练习

创建两个文件：

```text
led.h
led.c
```

`led.h` 中放：

```c
#ifndef LED_H
#define LED_H

#define LED_ON  1
#define LED_OFF 0

int led_init(void);
int led_set(int value);

#endif
```

`led.c` 中放：

```c
#include "led.h"

int led_init(void)
{
    return 0;
}

int led_set(int value)
{
    return 0;
}
```

要求：

1. 头文件必须有保护。
2. 函数声明放在 `.h`。
3. 函数实现放在 `.c`。
4. 不要在 `.h` 中定义普通全局变量。

## 12 小结

宏定义和头文件是 C 项目组织的基础。

需要重点记住：

1. 宏是预处理阶段的文本替换，不是变量，也不是函数。
2. 宏参数和整体表达式要加括号。
3. 简单常量和寄存器 bit 可以用宏表示。
4. 复杂逻辑优先考虑 `static inline` 函数。
5. 头文件用于放公共声明，不应随意放实现。
6. 全局变量在头文件中使用 `extern` 声明，在 `.c` 文件中定义。
7. 每个头文件都应该有头文件保护。

对嵌入式和 Linux 驱动开发来说，宏和头文件不仅是语法问题，更是代码可维护性问题。
