# QEMU 和 KVM 概述

## 1.1 虚拟化简介

### 1.1.1 虚拟化思想

计算机发展中虚拟化思想也在不断地发展。一些例子：

CPU执行的机器指令到汇编语言、C语言再到高级语言。
磁盘的磁道柱面概念到文件与目录。
TCP/IP协议栈使网卡透明，我们只需要提供数据和地址。
每一个进程都是对计算机的抽象，是一个状态机。
操作系统会提供API服务应用程序。

底层的资源通过空间的分割或者时间的分割，转换成一种简单易用的资源给上层使用，这就是虚拟化。

### 1.1.2 虚拟机简介

虚拟机是一个环境，一个可以完成用户指定的操作的机器。

最简单的虚拟机是进程。

模拟器是另一种形式的虚拟机，可以实现 ISA 的翻译。比如 QEMU 实现的用户态程序模拟。

高级语言的虚拟机在模拟器上更进一步，没有具体的运行在物理上的 ISA ，而是定义虚拟的字节码。
典型的高级语言虚拟机比如 JVM 虚拟机、Python 虚拟机。

进程、模拟器、高级语言的虚拟机都是提供指令的执行环境，而系统虚拟机是提供一个完整的系统环境。
系统级虚拟化中，管理全局物理资源的软件是虚拟机监控器 VMM。VMM 与 VM 的关系就像操作系统与进程一样。
系统虚拟化解决方案有：VMware Workstation、QEMU、Virtual Box

### 1.1.3 系统虚拟化历史

大型机时代的多用户使用需求。

普通PC也支持多个操作系统，多个用户。

云计算给租户提供虚拟化的资源。

Intel AMD 都在 CPU 硬件层面增加了虚拟化支持。
随着性能要求的提高，内存、外设也在硬件层面提供了虚拟化支持。

KVM是一个非常成功的VMM。

## 1.2 QEMU 和 KVM 架构介绍

### 1.2.1 QEMU 和 KVM 历史

QEMU 和 KVM 经常放在一起讨论，其实二者是可以解耦合的。

QEMU 可以完成用户程序的模拟和系统虚拟化模拟。
用户程序的模拟使一种平台编译的二进制可以在另一种平台上运行。TCG引擎翻译指令集。
系统虚拟化模拟是让虚拟机有自己的vCPU、芯片组、虚拟内存和外部设备，虚拟机中运行的操作系统看到的硬件视图和物理机是一样的。

KVM 本身是一个内核模块，导出了一系列接口到用户空间，用户空间可以使用这些接口创建虚拟机。
KVM 完成核心的CPU虚拟化和内存虚拟化，使用QEMU作为其用户态组件。

QEMU-KVM系统虚拟化软件已经成为云计算的基础。

### 1.2.2 QEMU 和 KVM 架构

#### QEMU 和 KVM 架构整体上分为3部分：

- VMX root 模式的应用层，QEMU 在这里实现平台的模拟。
- VMX root 模式的内核层，这里实现KVM内核驱动。
- VMX non-root 模式，这里运行虚拟机。

虚拟机在 VMX root 和 VMX non-root 模式下都有ring 0到ring 3四个特权级别。

QEMU 创建 CPU 线程来模拟虚拟机 CPU 执行流。在 QEMU 的进程地址空间中分配空间作为虚拟机的物理地址空间。QEMU
还会根据命令行参数为虚拟机创建虚拟设备。
QEMU 会在虚拟机运行过程中使用主线程监听事件，接收到事件后调用处理函数。

虚拟机本身有自己的应用层和内核层，都是运行在 VMX non-root 模式下的。
虚拟机的CPU是QEMU进程中的一个线程，会被直接调度到宿主机CPU上直接执行代码。
虚拟机操作系统有自己的页表，完成GVA->GPA，然后KVM中会有EPT，完成GPA->HPA的转换。
虚拟机中的设备是QEMU呈现的，在操作系统启动的时候进行设备枚举，加载相应的驱动。KVM会拦截虚拟机对设备的I/O请求，交给QEMU处理。

KVM驱动以杂项设备的方式存在在内核中。通过/dev/kvm导出一系列接口，QEMU使用这些接口创建虚拟机并控制。
KVM的重要工作是截取虚拟机产生的VM Exit事件进行处理。

#### QEMU 和 KVM 联合起来完成虚拟机各个组件的虚拟化

##### CPU虚拟化

QEMU会创建虚拟机CPU线程，初始化的时候会设置相应虚拟机寄存器的值，然后调用KVM接口，让虚拟机运行起来，直接在物理CPU上执行虚拟机的代码。
KVM会截获虚拟机中的敏感指令，CPU会从VMX non-root模式退出到KVM，这叫做VM Exit。虚拟机会退出到KVM进行处理，如果不能处理也可能转给QEMU处理。
KVM或者QEMU处理完后，CPU进入VMX non-root模式，运行虚拟机代码，这叫做VM Entry。

虚拟机会不停进行VM Exit和 VM Entry，CPU会加载对应的宿主机状态和虚拟机状态。
KVM使用VMCS数据结构来保存VM Exit和VM Entry，也就是物理机状态和虚拟机状态。

##### 内存虚拟化

QEMU在初始化的时候会调用KVM接口告知KVM虚拟机需要的物理内存。
QEMU在自己的进程地址空间通过mmap分配虚拟地址空间作为虚拟机的物理地址空间。
现在的CPU都支持EPT，这样虚拟机物理地址将直接转换为物理机物理地址GPA->HPA。

##### 设备虚拟化

QEMU最多的代码是设备模拟，设备模拟的本质是为虚拟机提供一个与物理接口完全一致的虚机接口。
虚拟机中的操作系统与设备的数据交互，可以由QEMU和KVM实现，也可以在设备后端实现。

设备模拟的方式有：
纯软件；
virtio，这是一种半虚拟化的方法，需要在虚拟机操作系统中安装virtio驱动；
设备直通，将物理设备直接挂载到虚拟机上，尽量减少I/O路径上QEMU KVM的参与。

SRIOV 单根输入输出虚拟化技术，使一个物理设备可以虚拟化出多个虚拟设备，然后直通到虚拟机中。

##### 中断虚拟化

中断是一个计算机系统不可或缺的组成部分。
操作系统通过写设备的I/O端口或者通过MMIO地址来与设备进行交互。
设备通过发送中断通知操作系统事件。

用户可以选择让QEMU KVM模拟全部的中断控制器。
QEMU和KVM一方面需要完成中断设备的模拟，另一方面需要模拟中断的请求。

中断控制器有：
单CPU的Intel 8259芯片、I/O APIC、LAPIC。

中断请求有：
ISA设备连接到8259的中断请求，PCIx设备的INTx请求，MSI 和 MSIX中断请求。

## 1.3 KVM API 使用实例

KVM通过一组ioctl向用户空间导出接口，这些接口用于虚拟机的创建、虚拟机内存的设置、虚拟机VCPU的创建与运行等。

KVM的ioctl可以分为3类：

- 系统全局的ioctl，作用对象是KVM模块，提供全局的配置项。
- 虚拟机相关的ioctl，作用对象是一台虚拟机，设置虚拟机的状态。
- 虚拟机VCPU相关的ioctl，作用对象是一个虚拟机VCPU，比如开始虚拟机VCPU的运行。

`kvm_userspace_memory_region` 用于定义和管理用户空间提供给虚拟机使用的内存区域。

每一个VCPU都有一个`struct kvm_run`结构用来在内核态KVM和用户态共享数据。

为了让VCPU运行起来，需要设置VCPU相关的寄存器，放在`kvm_regs`中。
