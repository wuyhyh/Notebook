# 03-中断机制基础

## 1 为什么要学习中断

在 Linux、驱动开发和嵌入式系统中，中断是非常基础的概念。

很多硬件事件都不是 CPU 主动轮询发现的，而是由设备通过中断通知 CPU。

例如：

```text
网卡收到数据包
串口收到字符
定时器到期
GPIO 电平变化
磁盘完成一次读写
PCIe 设备发出 MSI/MSI-X 中断
```

如果没有中断，CPU 只能不断查询设备状态，这会浪费大量 CPU 时间。

简单理解：

```text
中断 = 硬件或软件通知 CPU 有事情需要处理的一种机制
```

## 2 轮询和中断的区别

访问硬件事件通常有两种方式：

```text
轮询
中断
```

轮询是 CPU 主动反复查询设备状态。

例如：

```c
while (!(readl(status_reg) & READY))
    ;
```

这种方式简单，但浪费 CPU。

中断是设备在事件发生时主动通知 CPU。

例如：

```text
设备收到数据
        |
        v
触发中断
        |
        v
CPU 暂停当前执行流
        |
        v
执行中断处理程序
```

对比：

| 对比项 | 轮询 | 中断 |
|---|---|---|
| 触发方式 | CPU 主动查询 | 设备主动通知 |
| CPU 占用 | 可能很高 | 通常较低 |
| 响应方式 | 查询到状态后处理 | 中断触发后处理 |
| 适合场景 | 简单、短时间等待 | 异步事件、低功耗、外设通知 |

## 3 中断的基本流程

一个典型硬件中断的大致流程如下：

```text
外设发生事件
        |
        v
外设向中断控制器发出中断信号
        |
        v
中断控制器通知 CPU
        |
        v
CPU 暂停当前执行流
        |
        v
CPU 跳转到中断入口
        |
        v
内核执行中断处理程序
        |
        v
清除中断状态
        |
        v
返回原来的执行流
```

可以简单理解为：

```text
设备发信号，CPU 停下来处理，处理完再回去继续执行
```

## 4 中断控制器是什么

中断控制器负责管理多个中断源，并把中断请求分发给 CPU。

在 ARM64 系统中，常见中断控制器是 GIC。

GIC 是 Generic Interrupt Controller，中文通常叫通用中断控制器。

它的作用包括：

```text
接收外设中断信号
记录中断状态
设置中断优先级
决定中断发给哪个 CPU
支持中断使能和屏蔽
向 CPU 分发中断
```

简单理解：

```text
中断控制器 = 外设和 CPU 之间的中断管理中心
```

## 5 中断号是什么

系统中每个中断源通常会有一个编号，这个编号可以叫中断号。

例如：

```text
UART0 中断号
网卡中断号
定时器中断号
GPIO 中断号
```

在 Linux 中，驱动通常不应该硬编码中断号，而是从设备树或 ACPI 中获取中断资源。

设备树中可能会看到：

```dts
uart0: serial@28001000 {
    compatible = "arm,pl011";
    reg = <0x0 0x28001000 0x0 0x1000>;
    interrupts = <0 33 4>;
};
```

其中：

```text
interrupts = <0 33 4>;
```

描述了这个设备使用的中断资源。

具体每个字段的含义和中断控制器绑定文档有关，不同平台可能略有差异。

## 6 中断处理程序是什么

中断处理程序是内核中处理某个中断的函数。

在 Linux 驱动中，常见形式是：

```c
irqreturn_t xxx_irq_handler(int irq, void *dev_id)
{
    /* 读取设备状态 */
    /* 判断是否是本设备中断 */
    /* 清除中断 */
    /* 唤醒等待队列或调度后续处理 */

    return IRQ_HANDLED;
}
```

驱动需要向内核注册中断处理程序。

常见接口：

```c
request_irq(irq, xxx_irq_handler, flags, "xxx", dev);
```

释放中断时使用：

```c
free_irq(irq, dev);
```

简单理解：

```text
中断处理程序 = 中断发生后内核要执行的函数
```

## 7 中断处理程序中应该做什么

中断处理程序应该尽量短、快、明确。

通常适合做：

```text
读取中断状态
确认中断来源
清除设备中断状态
保存必要信息
唤醒等待队列
调度下半部处理
```

不适合做：

```text
长时间循环
复杂计算
大量打印日志
可能睡眠的操作
耗时 IO 操作
长时间持有锁
```

原因是中断处理程序运行时会影响系统响应能力。

如果中断处理函数执行太久，可能导致系统卡顿、丢中断或影响实时性。

## 8 为什么中断处理不能睡眠

Linux 中断处理程序运行在中断上下文。

中断上下文不是普通进程上下文，不能随便睡眠。

例如下面这些操作在普通中断处理程序中通常不合适：

```text
msleep()
mutex_lock() 可能睡眠
copy_to_user()
访问可能阻塞的文件系统接口
等待用户态响应
```

因为中断处理程序不是某个普通用户进程在执行，不能像进程一样被正常调度睡眠后再恢复。

简单记忆：

```text
中断上下文不能睡眠
进程上下文可以睡眠
```

## 9 上半部和下半部

为了让中断处理更快，Linux 常把中断处理拆成两部分：

```text
上半部
下半部
```

上半部就是直接响应硬件中断的处理函数。

它应该尽快完成关键工作，例如：

```text
确认中断
清除中断
记录状态
调度下半部
```

下半部用于处理比较耗时、不需要立刻完成的工作。

常见下半部机制包括：

```text
softirq
tasklet
workqueue
threaded IRQ
```

简单理解：

```text
上半部：快处理，别拖延
下半部：慢处理，后面做
```

## 10 workqueue 和 threaded IRQ

在实际驱动中，如果中断后续处理可能比较耗时，常用 workqueue 或 threaded IRQ。

workqueue 的特点是工作运行在内核线程中，因此可以睡眠。

例如适合做：

```text
较复杂的数据处理
可能等待的操作
需要使用 mutex 的操作
需要和其他内核子系统交互的操作
```

threaded IRQ 是把中断处理线程化。

常见接口是：

```c
request_threaded_irq(irq, top_handler, thread_fn, flags, name, dev);
```

其中：

```text
top_handler：中断上半部
thread_fn：中断线程处理函数
```

简单理解：

```text
threaded IRQ = 给中断处理安排一个内核线程来做较慢的工作
```

## 11 中断屏蔽和中断使能

中断可以被使能，也可以被屏蔽。

常见层级包括：

```text
设备内部中断使能位
中断控制器中的使能位
CPU 本地中断开关
Linux 内核中的 IRQ 状态管理
```

即使设备触发了中断，如果某一级被屏蔽，CPU 也可能收不到中断。

驱动开发中常见问题是：

```text
设备中断状态已经产生，但设备中断使能位没开
设备中断已经发出，但设备树中断号写错
中断控制器没有正确配置
驱动注册的 irq 不对
中断处理函数没有正确清除中断
```

所以调试中断时，要同时检查设备、中断控制器和驱动三层。

## 12 中断清除为什么重要

很多设备触发中断后，需要驱动清除设备内部的中断状态。

例如：

```text
写 1 清除某个中断状态位
读取某个寄存器后自动清除
先读状态寄存器再写清除寄存器
```

如果中断状态没有正确清除，可能出现：

```text
中断一直重复触发
系统不断进入中断处理函数
CPU 占用率异常升高
设备无法继续工作
```

所以中断处理程序通常必须做两件重要的事：

```text
读取中断状态
清除中断状态
```

具体清除方式要看芯片手册。

## 13 边沿触发和电平触发

中断触发方式常见有两类：

```text
边沿触发
电平触发
```

边沿触发关注信号变化瞬间。

例如：

```text
低电平到高电平
高电平到低电平
```

电平触发关注信号保持在某个电平。

例如：

```text
高电平有效
低电平有效
```

对比：

| 对比项 | 边沿触发 | 电平触发 |
|---|---|---|
| 触发条件 | 电平变化瞬间 | 保持某个有效电平 |
| 常见例子 | 上升沿、下降沿 | 高电平有效、低电平有效 |
| 清除重点 | 避免漏掉边沿 | 必须清除中断源，让电平恢复 |
| 调试现象 | 可能丢边沿 | 可能中断风暴 |

如果电平触发中断没有清除设备中断源，中断线会一直保持有效，CPU 会反复进入中断。

## 14 中断风暴是什么

中断风暴是指系统短时间内收到大量中断，导致 CPU 被中断处理占满。

常见原因：

```text
中断状态没有正确清除
设备异常持续触发中断
中断触发类型配置错误
驱动误判中断来源
硬件信号不稳定
```

现象可能包括：

```text
系统卡顿
CPU 占用率很高
日志疯狂打印
设备响应异常
/proc/interrupts 中某个中断计数快速增长
```

调试时可以查看：

```bash
cat /proc/interrupts
```

如果某个中断计数疯狂增长，就要重点检查对应设备和驱动。

## 15 共享中断

有些平台或总线上，多个设备可能共享同一个 IRQ。

注册共享中断时通常会使用：

```c
IRQF_SHARED
```

共享中断处理程序中必须判断：

```text
这个中断是不是本设备产生的
```

如果不是本设备产生的，应该返回：

```c
return IRQ_NONE;
```

如果确认是本设备产生的，处理后返回：

```c
return IRQ_HANDLED;
```

简单理解：

```text
共享中断下，每个驱动都要先确认是不是自己的中断
```

## 16 Linux 中如何查看中断

查看系统中断计数：

```bash
cat /proc/interrupts
```

这个文件可以看到每个 IRQ 在各个 CPU 上的触发次数。

示例形式：

```text
           CPU0       CPU1
 33:       1000          0     GICv3  33 Level     uart-pl011
 45:       2000       1500     GICv3  45 Level     eth0
```

重点观察：

```text
IRQ 编号
每个 CPU 上的中断计数
中断控制器名称
触发类型
设备名称
```

查看某个关键词：

```bash
cat /proc/interrupts | grep -i eth
cat /proc/interrupts | grep -i uart
```

持续观察中断变化：

```bash
watch -n 1 cat /proc/interrupts
```

## 17 驱动中常见中断相关接口

常见接口包括：

```c
request_irq()
free_irq()
request_threaded_irq()
enable_irq()
disable_irq()
disable_irq_nosync()
synchronize_irq()
```

简单说明：

| 接口 | 作用 |
|---|---|
| request_irq | 注册中断处理函数 |
| free_irq | 释放中断 |
| request_threaded_irq | 注册线程化中断 |
| enable_irq | 使能某个 IRQ |
| disable_irq | 禁用某个 IRQ，并等待正在执行的处理结束 |
| disable_irq_nosync | 禁用某个 IRQ，但不等待当前处理结束 |
| synchronize_irq | 等待某个 IRQ 当前处理完成 |

初学时重点掌握：

```text
request_irq
free_irq
request_threaded_irq
/proc/interrupts
```

## 18 简单实验

### 18.1 查看系统中断

执行：

```bash
cat /proc/interrupts
```

观察系统中有哪些中断。

重点看这些信息：

```text
中断号
CPU0/CPU1/... 上的计数
中断控制器
触发类型
设备名称
```

### 18.2 观察网卡中断变化

先查看网卡相关中断：

```bash
cat /proc/interrupts | grep -i eth
```

然后产生一些网络流量，例如：

```bash
ping 网关地址
```

再查看：

```bash
cat /proc/interrupts | grep -i eth
```

如果网卡使用中断，相关计数可能会增加。

### 18.3 观察串口中断变化

查看串口相关中断：

```bash
cat /proc/interrupts | grep -i uart
```

如果通过串口输入字符，再观察计数是否变化：

```bash
cat /proc/interrupts | grep -i uart
```

这可以帮助理解外设事件和中断计数之间的关系。

## 19 常见理解误区

### 19.1 中断不是普通函数调用

中断处理函数不是由应用程序直接调用的普通函数。

它是硬件事件触发后，由 CPU 和内核中断框架进入执行的。

### 19.2 中断处理不是越多越好

中断可以提高响应能力，但中断太频繁也会带来开销。

高性能网卡等设备常会使用中断合并、NAPI 等机制减少中断压力。

### 19.3 中断处理函数不能随便睡眠

普通硬中断处理程序运行在中断上下文，不能执行可能睡眠的操作。

如果确实需要睡眠，应考虑 workqueue 或 threaded IRQ。

### 19.4 设备触发中断后，驱动通常要清中断

很多设备需要驱动清除中断状态。

如果不清除，中断可能会反复触发，形成中断风暴。

### 19.5 /proc/interrupts 很重要

调试驱动时，`/proc/interrupts` 是非常重要的观察入口。

它可以帮助判断中断是否触发、触发次数是否异常、是否分布在不同 CPU 上。

## 20 总结

中断机制可以这样理解：

```text
设备发生事件
设备通知中断控制器
中断控制器通知 CPU
CPU 进入内核中断处理程序
驱动处理事件并清除中断
CPU 返回原来的执行流
```

一句话总结：

```text
中断是设备通知 CPU 处理异步事件的机制，是操作系统和驱动程序响应硬件事件的基础。
```

驱动开发中要特别记住：

```text
中断处理要尽量短
硬中断上下文不能睡眠
需要正确清除设备中断状态
复杂工作应放到下半部、workqueue 或 threaded IRQ
调试中断时重点查看 /proc/interrupts
```
