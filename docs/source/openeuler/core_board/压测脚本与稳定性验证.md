# Kernel 压力测试用例（arm64 自研板 / NVMe / ext4 / Linux 5.10）

> 适用环境：openEuler Server（arm64），自编译 Linux 5.10
> 目标：为自研板快速发现并复现 **NVMe/PCIe/IRQ/RCU/ext4(jbd2)/内存回收** 等关键稳定性问题，并提供回归门禁（smoke 30min）。

---

## 1. 测试范围与判定标准

### 1.1 覆盖子系统

* CPU/调度/上下文切换（soft lockup / RCU stall / task blocked）
* 内存分配与页回收（kswapd 抖动、OOM 边缘）
* NVMe/块层/blk-mq（timeout、controller reset、AER、队列深度临界）
* ext4/jbd2（元数据风暴、fsync/flush/journal abort）
* 网络（可选：软中断与 IO 竞争）

### 1.2 Fail 条件（任意一条命中即 Fail）

* kernel panic / reboot / hang / watchdog reset
* `dmesg/journalctl -k` 出现：

    * `BUG:` `Oops` `WARNING:`
    * `soft lockup` `hard lockup`
    * `RCU stall`
    * `INFO: task ... blocked for more than` / 大量 `D` 状态进程
    * `nvme.*timeout` `controller reset` / `pcie.*AER`
    * `EXT4-fs error` `JBD2` `Aborted journal` / `Buffer I/O error`

### 1.3 Pass 条件（最低要求）

* 用例时长内系统可用（SSH 不断、可执行命令）
* 用例结束后 10~30 分钟内负载、内存、IO 回落到合理水平
* 日志无 Fail 关键字

## 2. 测试环境信息（固定记录）

你的环境（示例）：

* CPU：Phytium S2500，8C，1 socket，最高 2.3GHz
* NUMA：1 node
* NVMe：`/dev/nvme0`，测试分区：`/dev/nvme0n1p3`
* FS：ext4，挂载点：`/mnt/test`
* NVMe `mdts=7`：通常意味着单命令传输上限约 **512KiB**（以 4K 页推算），顺序吞吐建议 `bs=512K`

> ⚠️ 强烈要求：`/dev/nvme0n1p3` 为**专用测试分区**，避免覆盖系统数据。



## 5. 全程监控（建议测试期间一直开着）

```bash
nohup sh -c '
while true; do
  date
  uptime
  echo "---- vmstat ----"; vmstat 1 5
  echo "---- mpstat ----"; mpstat -P ALL 1 5
  echo "---- iostat ----"; iostat -x 1 5
  echo "---- interrupts(top) ----"
  awk "{s=0; for(i=2;i<=NF;i++) s+=\$i; print s, \$1}" /proc/interrupts 2>/dev/null | sort -nr | head -n 15
  echo "---- nvme smart ----"
  nvme smart-log /dev/nvme0 2>/dev/null | egrep -i "temperature|media|error|critical|warning" || true
  echo "========"
  sleep 10
done
' > logs/monitor.txt 2>&1 &
echo $! > logs/monitor.pid
```

停止监控：

```bash
kill "$(cat logs/monitor.pid)" 2>/dev/null || true
```

## 6. 回归门禁：30 分钟 Smoke（必跑）

> 目标：30 分钟内尽可能把常见雷（NVMe/PCIe、ext4、RCU/lockup）炸出来
> 组成：20 分钟 NVMe 随机 4K + 10 分钟 ext4 元数据风暴（总计 30min）

### SMOKE-IO-4K（20min，最关键）

```bash
fio --name=smoke_rand4k --directory=/mnt/test --rw=randrw --rwmixread=70 \
    --bs=4K --iodepth=64 --numjobs=8 --size=8G \
    --time_based --runtime=20m --direct=1 --group_reporting \
    2>&1 | tee logs/smoke_01_rand4k.txt

dmesg -T | egrep -i "nvme|pcie|aer|timeout|reset|abort|fatal|error|IOMMU|SMMU|RCU stall|lockup" | tail -n 200 \
  | tee logs/smoke_01_rand4k.dmesg.txt
```

### SMOKE-EXT4-META（10min）

```bash
mkdir -p /mnt/test/meta && cd /mnt/test/meta
rm -rf ./*

# 控制在 10min 左右：并发 8 组、每组 15000 次
for t in $(seq 1 8); do
  (mkdir -p d$t && cd d$t &&
   for i in $(seq 1 15000); do
     echo x > f$i
     mv f$i g$i
     rm -f g$i
   done) &
done
wait
sync

dmesg -T | egrep -i "EXT4-fs error|JBD2|Aborted journal|Buffer I/O error" | tail -n 200 \
  | tee logs/smoke_02_ext4meta.dmesg.txt
```

## 7. 专项用例（定位与复现）

### 7.1 NVMe/PCIe：队列深度临界点扫描（定位神器）

```bash
for depth in 1 4 8 16 32 64 128; do
  echo "=== iodepth=$depth ===" | tee -a logs/tc_nvme_qdscan.txt
  fio --name=qd$depth --directory=/mnt/test --rw=randwrite --bs=4K \
      --iodepth=$depth --numjobs=4 --size=4G --time_based --runtime=8m \
      --direct=1 --group_reporting 2>&1 | tee -a logs/tc_nvme_qdscan.txt

  dmesg -T | egrep -i "nvme|timeout|reset|pcie|aer|fatal|error" | tail -n 80 \
    | tee -a logs/tc_nvme_qdscan.dmesg.txt
done
```

### 7.2 NVMe 顺序吞吐（贴 mdts=7：bs=512K）

```bash
fio --name=seqrw_512k --directory=/mnt/test --rw=readwrite --bs=512K \
    --iodepth=32 --numjobs=2 --size=20G --time_based --runtime=30m \
    --direct=1 --group_reporting 2>&1 | tee logs/tc_nvme_seqrw_512k.txt
```

### 7.3 ext4：fsync 路径（jbd2/flush）

```bash
fio --name=ext4_fsync --directory=/mnt/test --rw=write --bs=4K \
    --ioengine=sync --fdatasync=1 --size=8G \
    --time_based --runtime=30m --group_reporting \
    2>&1 | tee logs/tc_ext4_fsync.txt

dmesg -T | egrep -i "EXT4-fs error|JBD2|Aborted journal|I/O error|nvme|timeout" | tail -n 200 \
  | tee logs/tc_ext4_fsync.dmesg.txt
```

### 7.4 调度/RCU：switch+futex 与 IO 叠加（易触发 stall）

```bash
stress-ng --switch 2 --futex 4 --timeout 30m --metrics-brief \
  2>&1 | tee logs/tc_sched_switch_futex.txt &
P1=$!

fio --name=rand4k --directory=/mnt/test --rw=randrw --rwmixread=70 \
    --bs=4K --iodepth=64 --numjobs=4 --size=8G \
    --time_based --runtime=30m --direct=1 --group_reporting \
    2>&1 | tee logs/tc_sched_rand4k.txt

wait $P1
dmesg -T | egrep -i "RCU stall|soft lockup|hard lockup|task blocked|nvme|pcie|aer|EXT4-fs error|JBD2" | tail -n 250 \
  | tee logs/tc_sched_mix.dmesg.txt
```

## 9. 出问题时的快速归因提示（自研板常用）

### 9.1 更像 PCIe/NVMe 的信号

* `nvme timeout` / `controller reset` / `I/O error`
* `pcie AER` / `Fatal error` / 链路训练异常
* 伴随 IO 时系统卡死或大量任务进入 `D` 状态

优先动作：

* 看 `dmesg.pcie_nvme.*`、`nvme smart-log`（介质错误/错误计数/温度）
* 看 `/proc/interrupts`：NVMe 中断是否长期集中在单核导致饥饿

### 9.2 更像 ext4/jbd2 的信号

* `EXT4-fs error` / `Aborted journal` / `JBD2` 大量异常
* 常常“根因”仍是底层 IO（所以要先排 NVMe 错误）

### 9.3 更像 RCU/调度/锁的信号

* `RCU stall` / `soft lockup` / `task blocked`
* 特别容易在 “IO + switch/futex” 混合场景出现

---

## 10. 建议的执行顺序（最省时间）

1. 每次改内核先跑：**Smoke 30min（第 6 章）**
2. 若 Smoke Fail：优先跑 **QD 扫描（7.1）** 定位最短复现参数
3. 若 IO 不报错但仍卡：跑 **调度混合（7.4）**
4. ext4 报错：先看 NVMe 是否已有 timeout/reset，再看 ext4/jbd2
