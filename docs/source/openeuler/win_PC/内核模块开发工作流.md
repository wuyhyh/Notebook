# 内核模块开发工作流 (WSL + CLion)

`oot-modules` 是一个用于 **out-of-tree (OOT) Linux 内核模块**的统一工作区，目标是：

- 与 `openeuler-fusion` 的内核开发工作流集成：按内核分支自动映射 `O=` 输出目录
- 在 WSL/Linux 上方便地 **编译 OOT 模块**（Kbuild/KDIR/M= 方式）
- （WSL 可选）自动生成 `compile_commands.json`，让 **CLion/clangd 立即可索引/跳转/补全**
- 支持复杂模块：多 `.c`、自定义头文件、子目录源码（递归扫描 `.cmd`）

> 核心思路：  
> **Kbuild 在编译时生成 `.cmd` 文件**，里面记录每个 `.c` 的真实编译命令；  
> `tools/ccdb.sh` 递归解析这些 `.cmd`，生成 `compile_commands.json`（仅开发机需要）。

---

## 1. 目录布局要求

推荐（也是本仓库默认假设）：

```

~/kernel/
├── openeuler-fusion/     # 内核源码仓库（你维护）
└── oot-modules/          # 本仓库（OOT 模块集合）

````

即：`oot-modules` 与 `openeuler-fusion` **并排同级**。

---

## 2. 环境变量与分支映射（env.sh）

在 `oot-modules/` 根目录：

```bash
source env.sh
````

`env.sh` 会导出：

* `KSRCDIR`：指向 `../openeuler-fusion`
* `BR`：`openeuler-fusion` 当前分支名
* `O`：`$HOME/build/openeuler-fusion/$BR`
* `ARCH`：默认 `arm64`（如需覆盖可在 source 前 export）

你也可以手动覆盖：

```bash
export KSRCDIR=/path/to/openeuler-fusion
export O=/path/to/kernel-build-output
export ARCH=arm64
source env.sh
```

---

## 3. 内核侧前置条件（fusion 模式必需）

fusion 模式（使用 `openeuler-fusion + O=`）编译 OOT 模块，依赖 `O=` 中的生成头文件（例如 `include/generated/autoconf.h`）。

在 `openeuler-fusion/` 中至少确保执行过（以你的 `O=` 为准）：

```bash
make O="$O" ARCH="$ARCH" olddefconfig
make O="$O" ARCH="$ARCH" prepare modules_prepare
```

---

## 4. 编译模块：两种目标（WSL/开发机 + 商用板通用）

本仓库每个模块的 Makefile 统一支持两种构建目标：

### 4.1 fusion 模式（默认）

适用于：跟随 `openeuler-fusion` 某分支构建 OOT 模块。

```bash
cd ~/kernel/oot-modules
source env.sh

make -C hello -j"$(nproc)"
```

### 4.2 running 模式（板子上最常用）

适用于：编译 **当前运行内核** 可加载的模块（匹配 `uname -r`）。

```bash
cd ~/kernel/oot-modules

KDIR=/lib/modules/$(uname -r)/build
make -C hello KDIR="$KDIR" -j"$(nproc)"
```

> 注意：running 模式要求 `/lib/modules/$(uname -r)/build` 存在且完整。
> 若缺失，需要安装与运行内核匹配的 `kernel-devel/kernel-headers`（发行版包名以仓库为准）。

## 5. （可选）生成索引数据库（仅 WSL/开发机需要）

在 WSL/开发机上，为 CLion/clangd 生成 `compile_commands.json`：

```bash
cd ~/kernel/oot-modules
source env.sh

# 先确保模块至少编译过一次（生成 .cmd）
make -C kmsg_deep -j"$(nproc)"

# 生成 compile_commands.json（递归扫描 .cmd）
tools/ccdb.sh kmsg_deep
```

也支持批量：

```bash
tools/ccdb.sh hello kmsg_ping kmsg_deep
```

生成结果：

* `<module>/compile_commands.json`

IDE 使用：

* CLion：打开模块目录或整个仓库，自动识别 `compile_commands.json`
* clangd：同理（VSCode/CLion/其他支持 compilation database 的 IDE 均可）

## 6. 模块组织规范（建议）

### 6.1 单文件模块

目录内 `Makefile` 典型写法：

```make
obj-m += hello.o
```

### 6.2 多文件模块

例如 `kmsg_ping`：

```make
obj-m += kmsg_ping.o
kmsg_ping-y := src/main.o src/ping.o
ccflags-y += -I$(src)/include
```

### 6.3 深层子目录源码（验证递归）

例如 `kmsg_deep`：

```make
obj-m += kmsg_deep.o
kmsg_deep-y := src/core/main.o src/core/emit.o
ccflags-y += -I$(src)/include
```

> 结论：
> 只要 Kbuild 实际编译产生 `.cmd`，`tools/ccdb.sh` 就能生成索引数据库。
> 模块越复杂越应该让 Kbuild 管理编译参数（`ccflags-y` / `EXTRA_CFLAGS` 等）。

## 7. 常见问题

### 7.1 `tools/ccdb.sh` 报 “no .*.cmd found”

说明模块还没编译过（或 clean 了）。先：

```bash
make -C <module>
```

### 7.2 IDE 里大量红线/无法解析

通常是：

* `compile_commands.json` 没生成或没被 IDE 识别
* 内核侧 `O=` 没做 `prepare modules_prepare`（缺 generated headers）
* 切换内核分支后 `O=` 变化但未重新生成 compile db

解决顺序：

```bash
make -C <module>
tools/ccdb.sh <module>
# 然后在 IDE 里 Reload compilation database / 重开工程
```

### 7.3 切分支后的流程

`openeuler-fusion` 切分支 → `O=` 跟随变化。建议：

```bash
source env.sh
make -C <module>
tools/ccdb.sh <module>
```

### 7.4 running 模式编译失败：找不到 build 目录

检查：

```bash
ls -l /lib/modules/$(uname -r)/build
```

若不存在，需要安装内核开发包/headers（与运行内核版本匹配）。

### 7.5 fusion 模式编译正常但 insmod 失败

这是正常现象：fusion 模式生成的模块是否能加载到“当前运行内核”，取决于两者是否完全一致（vermagic/CONFIG/符号版本/签名策略等）。
要保证可加载，请用 running 模式编译。
