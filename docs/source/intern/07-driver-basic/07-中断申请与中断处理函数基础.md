# 07-中断申请与中断处理函数基础

## 1. 文档目标

本文用于整理 Linux 驱动开发中最基础的中断处理流程，重点服务于实习生理解：

```text
设备树描述 interrupt
  ↓
platform driver 在 probe() 中获取 irq
  ↓
驱动申请中断处理函数
  ↓
硬件触发中断
  ↓
内核调用中断处理函数
  ↓
驱动读取状态并清除中断
```

完成本文后，应该能够理解：

1. 中断在驱动中的作用；
2. 设备树中 `interrupts` 的基本含义；
3. `platform_get_irq()` 如何获取中断号；
4. `devm_request_irq()` 如何注册中断处理函数；
5. 中断处理函数的基本写法；
6. `IRQ_HANDLED` 和 `IRQ_NONE` 的区别；
7. 为什么中断处理函数中不能随便睡眠；
8. 如何通过 `/proc/interrupts` 和 `dmesg` 调试中断问题。

本文只讲最基础的中断申请和处理，不深入展开 threaded IRQ、softirq、tasklet、workqueue、MSI、NAPI 等高级主题。

---

## 2. 为什么驱动需要中断

硬件设备通常有两种方式通知 CPU：

```text
轮询
中断
```

轮询方式是 CPU 主动不断检查设备状态：

```text
CPU 不断读取状态寄存器
  ↓
判断设备是否有事件发生
```

这种方式简单，但浪费 CPU。

中断方式是设备主动通知 CPU：

```text
设备发生事件
  ↓
设备向中断控制器发出中断信号
  ↓
中断控制器通知 CPU
  ↓
Linux 内核调用对应驱动的中断处理函数
```

例如：

| 设备 | 中断触发场景 |
|---|---|
| UART | 收到数据 |
| 网卡 | 收到网络包或发送完成 |
| GPIO | 引脚电平变化 |
| I2C 控制器 | 传输完成或错误 |
| SPI 控制器 | FIFO 空或传输完成 |
| DMA 控制器 | DMA 传输完成 |
| 自定义外设 | 状态变化或任务完成 |

所以，中断是驱动开发中非常核心的机制。

---

## 3. 中断处理的基本流程

一个典型 platform driver 的中断处理流程如下：

```text
DTS 中描述 interrupts
  ↓
probe() 中调用 platform_get_irq()
  ↓
得到 Linux irq number
  ↓
调用 devm_request_irq()
  ↓
注册 irq handler
  ↓
硬件触发中断
  ↓
内核调用 irq handler
  ↓
handler 读取中断状态寄存器
  ↓
判断是否为本设备中断
  ↓
清除中断状态
  ↓
返回 IRQ_HANDLED
```

核心代码通常是：

```c
irq = platform_get_irq(pdev, 0);
if (irq < 0)
	return irq;

ret = devm_request_irq(dev, irq, my_irq_handler, 0,
		       dev_name(dev), priv);
if (ret)
	return ret;
```

## 4. 设备树中的interrupts属性

在设备树中，中断通常通过 `interrupts` 或 `interrupts-extended` 描述。

一个常见 ARM GIC 示例：

```dts
my_demo@28000000 {
	compatible = "training,my-irq-demo";
	reg = <0x0 0x28000000 0x0 0x1000>;
	interrupts = <0 45 4>;
	status = "okay";
};
```

这里：

```dts
interrupts = <0 45 4>;
```

具体含义依赖中断控制器的 binding。对于常见 ARM GIC，通常可以粗略理解为：

| cell | 含义 |
|---|---|
| 第 1 个 | 中断类型，例如 SPI 或 PPI |
| 第 2 个 | 硬件中断号 |
| 第 3 个 | 触发类型，例如边沿触发或电平触发 |

对初学者来说，暂时不要求死记每个数字的含义。更重要的是理解：

```text
interrupts 是设备树告诉内核：
这个设备连接到了哪个中断控制器的哪一路中断。
```

实际项目中，必须根据 SoC 文档、设备树绑定文档和已有 DTS 示例来填写。

## 5. interrupt-parent

如果设备节点所在的父节点没有默认中断控制器，或者需要明确指定中断控制器，可以使用：

```dts
interrupt-parent = <&gic>;
```

示例：

```dts
my_demo@28000000 {
	compatible = "training,my-irq-demo";
	reg = <0x0 0x28000000 0x0 0x1000>;
	interrupt-parent = <&gic>;
	interrupts = <0 45 4>;
	status = "okay";
};
```

可以简单理解为：

```text
interrupt-parent 指明这个设备的中断接到哪个中断控制器。
interrupts 描述具体是哪一路中断以及触发方式。
```

很多 SoC 的外设节点不直接写 `interrupt-parent`，是因为上级节点已经指定了默认中断控制器。

## 6. Linux irq number和硬件中断号

初学者容易混淆：

```text
硬件中断号
Linux irq number
```

设备树中描述的是硬件连接关系。Linux 启动后，会把硬件中断映射成内核使用的 irq number。

驱动中使用的是：

```c
int irq;
```

这个 `irq` 是 Linux 内核中的中断号，不一定等于设备树里的原始硬件中断号。

例如：

```c
irq = platform_get_irq(pdev, 0);
```

返回的是 Linux irq number。

驱动后续申请中断时，也使用这个 irq：

```c
devm_request_irq(dev, irq, my_irq_handler, 0, dev_name(dev), priv);
```

所以，驱动中不要自己硬编码硬件中断号，而应该从设备树或总线资源中获取。

---

## 7. platform_get_irq

`platform_get_irq()` 用于从 platform device 中获取中断资源。

常见写法：

```c
irq = platform_get_irq(pdev, 0);
if (irq < 0) {
	dev_err(dev, "failed to get irq: %d\n", irq);
	return irq;
}
```

其中：

```c
platform_get_irq(pdev, 0)
```

表示获取第 0 个中断资源。

如果设备有多个中断，可能会使用：

```c
irq0 = platform_get_irq(pdev, 0);
irq1 = platform_get_irq(pdev, 1);
```

例如一个设备可能有：

```text
rx interrupt
tx interrupt
error interrupt
```

对入门实验来说，先掌握一个中断即可。

---

## 8. request_irq和devm_request_irq

Linux 中申请中断常见接口有：

```c
request_irq()
devm_request_irq()
```

传统 `request_irq()` 示例：

```c
ret = request_irq(irq, my_irq_handler, 0, "my_demo", priv);
if (ret)
	return ret;
```

释放时需要：

```c
free_irq(irq, priv);
```

更推荐入门阶段使用 `devm_request_irq()`：

```c
ret = devm_request_irq(dev, irq, my_irq_handler, 0,
		       dev_name(dev), priv);
if (ret) {
	dev_err(dev, "failed to request irq: %d\n", ret);
	return ret;
}
```

`devm_request_irq()` 的好处是：

```text
设备解绑或 probe 失败时，中断资源会自动释放。
```

这可以减少 `remove()` 中的手动清理代码。

---

## 9. devm_request_irq参数说明

函数原型可以简化理解为：

```c
int devm_request_irq(struct device *dev,
		     unsigned int irq,
		     irq_handler_t handler,
		     unsigned long irqflags,
		     const char *devname,
		     void *dev_id);
```

常用参数说明：

| 参数 | 说明 |
|---|---|
| `dev` | 当前设备对象 |
| `irq` | Linux irq number |
| `handler` | 中断处理函数 |
| `irqflags` | 中断标志 |
| `devname` | 中断名称，会显示在 `/proc/interrupts` |
| `dev_id` | 传给中断处理函数的私有数据 |

常见写法：

```c
ret = devm_request_irq(dev, irq, my_irq_handler, 0,
		       dev_name(dev), priv);
```

这里的 `priv` 会在中断处理函数中通过 `void *data` 传回来。

---

## 10. 中断处理函数的基本形式

中断处理函数通常写成：

```c
static irqreturn_t my_irq_handler(int irq, void *data)
{
	struct my_demo_dev *priv = data;
	u32 status;

	status = readl(priv->base + MY_REG_IRQ_STATUS);
	if (!status)
		return IRQ_NONE;

	writel(status, priv->base + MY_REG_IRQ_CLEAR);

	return IRQ_HANDLED;
}
```

函数参数：

| 参数 | 含义 |
|---|---|
| `irq` | 触发的 Linux irq number |
| `data` | 注册中断时传入的私有数据 |

返回值：

| 返回值 | 含义 |
|---|---|
| `IRQ_HANDLED` | 这是本设备的中断，并已处理 |
| `IRQ_NONE` | 不是本设备的中断，或者没有可处理状态 |

---

## 11. 为什么要读取中断状态寄存器

很多硬件设备触发中断后，驱动需要读取中断状态寄存器，判断具体原因。

例如：

```c
#define MY_REG_IRQ_STATUS	0x10
#define MY_REG_IRQ_CLEAR	0x14

#define MY_IRQ_RX_DONE		BIT(0)
#define MY_IRQ_TX_DONE		BIT(1)
#define MY_IRQ_ERROR		BIT(2)
```

中断处理函数中：

```c
status = readl(priv->base + MY_REG_IRQ_STATUS);

if (status & MY_IRQ_RX_DONE)
	dev_info(priv->dev, "rx done\n");

if (status & MY_IRQ_TX_DONE)
	dev_info(priv->dev, "tx done\n");

if (status & MY_IRQ_ERROR)
	dev_err(priv->dev, "hardware error\n");
```

如果中断线是共享的，或者硬件状态没有置位，驱动应该返回：

```c
IRQ_NONE
```

如果确认是本设备中断并处理了，返回：

```c
IRQ_HANDLED
```

---

## 12. 为什么要清除中断

很多硬件中断需要驱动主动清除状态位。

如果不清除，中断可能会一直保持有效，导致：

```text
中断处理函数不断被调用
CPU 占用率升高
系统卡死
日志疯狂刷屏
```

常见清中断方式包括：

```text
写 1 清除，也就是 W1C
写 0 清除
读某个寄存器后自动清除
写特定值到 clear 寄存器
```

具体方式必须看硬件手册。

常见 W1C 示例：

```c
status = readl(priv->base + MY_REG_IRQ_STATUS);
writel(status, priv->base + MY_REG_IRQ_STATUS);
```

也有硬件使用单独 clear 寄存器：

```c
status = readl(priv->base + MY_REG_IRQ_STATUS);
writel(status, priv->base + MY_REG_IRQ_CLEAR);
```

不要凭感觉清中断，必须按照手册。

---

## 13. 中断处理函数中不能做什么

普通硬中断处理函数运行在中断上下文中，有很多限制。

一般不要在普通中断处理函数中做：

```text
1. 睡眠
2. 调用可能睡眠的函数
3. 长时间循环
4. 大量打印日志
5. 复杂内存分配
6. 等待 mutex
7. 执行耗时的数据处理
8. 访问用户空间
```

例如，不要在普通中断处理函数中调用：

```c
msleep(10);
mutex_lock(&lock);
copy_to_user(...);
```

中断处理函数应该尽量短：

```text
快速读取状态
快速清除中断
记录必要信息
唤醒后续处理
尽快返回
```

复杂工作可以放到 threaded IRQ、workqueue 或其他下半部机制中。入门阶段先知道：普通中断处理函数不能当普通函数随便写。

## 14. 中断标志irqflags

`devm_request_irq()` 的第四个参数是 `irqflags`。

最简单情况下可以填：

```c
0
```

例如：

```c
ret = devm_request_irq(dev, irq, my_irq_handler, 0,
		       dev_name(dev), priv);
```

常见标志包括：

| 标志 | 说明 |
|---|---|
| `IRQF_SHARED` | 允许共享中断线 |
| `IRQF_TRIGGER_RISING` | 上升沿触发 |
| `IRQF_TRIGGER_FALLING` | 下降沿触发 |
| `IRQF_TRIGGER_HIGH` | 高电平触发 |
| `IRQF_TRIGGER_LOW` | 低电平触发 |
| `IRQF_ONESHOT` | threaded IRQ 常用 |

如果设备树已经描述了触发方式，一般不需要在驱动里重复硬编码触发类型。  
入门阶段建议先使用：

```c
0
```

让中断控制器按照设备树中的配置处理。

---

## 15. 共享中断

有些平台上多个设备可能共享同一条中断线。这时申请中断时需要：

```c
IRQF_SHARED
```

示例：

```c
ret = devm_request_irq(dev, irq, my_irq_handler, IRQF_SHARED,
		       dev_name(dev), priv);
```

使用共享中断时，中断处理函数必须检查是否真的是自己的设备触发的中断。

例如：

```c
status = readl(priv->base + MY_REG_IRQ_STATUS);
if (!status)
	return IRQ_NONE;
```

如果不是自己的中断，应该返回：

```c
IRQ_NONE
```

如果是自己的中断，处理后返回：

```c
IRQ_HANDLED
```

对初学者来说，先掌握非共享中断；看到共享中断时要特别注意状态判断。

## 16. 最小实验代码

下面是一个最小 platform IRQ demo 示例。

注意：该代码假设设备有 MMIO 寄存器，并且中断状态和清除寄存器偏移如下。实际项目必须按照硬件手册修改。

```c
#include <linux/module.h>
#include <linux/platform_device.h>
#include <linux/of.h>
#include <linux/io.h>
#include <linux/interrupt.h>

#define MY_REG_IRQ_STATUS	0x10
#define MY_REG_IRQ_CLEAR	0x14

#define MY_IRQ_DONE		BIT(0)
#define MY_IRQ_ERROR		BIT(1)

struct my_irq_demo {
	struct device *dev;
	void __iomem *base;
	int irq;
};

static irqreturn_t my_irq_demo_handler(int irq, void *data)
{
	struct my_irq_demo *priv = data;
	u32 status;

	status = readl(priv->base + MY_REG_IRQ_STATUS);
	if (!status)
		return IRQ_NONE;

	if (status & MY_IRQ_DONE)
		dev_info(priv->dev, "irq done event\n");

	if (status & MY_IRQ_ERROR)
		dev_err(priv->dev, "irq error event\n");

	/* 按硬件手册要求清除中断。这里假设使用独立 clear 寄存器。 */
	writel(status, priv->base + MY_REG_IRQ_CLEAR);

	return IRQ_HANDLED;
}

static int my_irq_demo_probe(struct platform_device *pdev)
{
	struct device *dev = &pdev->dev;
	struct my_irq_demo *priv;
	int ret;

	dev_info(dev, "probe start\n");

	priv = devm_kzalloc(dev, sizeof(*priv), GFP_KERNEL);
	if (!priv)
		return -ENOMEM;

	priv->dev = dev;

	priv->base = devm_platform_ioremap_resource(pdev, 0);
	if (IS_ERR(priv->base)) {
		dev_err(dev, "failed to map registers\n");
		return PTR_ERR(priv->base);
	}

	priv->irq = platform_get_irq(pdev, 0);
	if (priv->irq < 0) {
		dev_err(dev, "failed to get irq: %d\n", priv->irq);
		return priv->irq;
	}

	ret = devm_request_irq(dev, priv->irq, my_irq_demo_handler, 0,
			       dev_name(dev), priv);
	if (ret) {
		dev_err(dev, "failed to request irq: %d\n", ret);
		return ret;
	}

	platform_set_drvdata(pdev, priv);

	dev_info(dev, "irq = %d\n", priv->irq);
	dev_info(dev, "probe done\n");

	return 0;
}

static int my_irq_demo_remove(struct platform_device *pdev)
{
	dev_info(&pdev->dev, "remove called\n");
	return 0;
}

static const struct of_device_id my_irq_demo_of_match[] = {
	{ .compatible = "training,my-irq-demo" },
	{ }
};

MODULE_DEVICE_TABLE(of, my_irq_demo_of_match);

static struct platform_driver my_irq_demo_driver = {
	.probe = my_irq_demo_probe,
	.remove = my_irq_demo_remove,
	.driver = {
		.name = "my_irq_demo",
		.of_match_table = my_irq_demo_of_match,
	},
};

module_platform_driver(my_irq_demo_driver);

MODULE_LICENSE("GPL");
MODULE_AUTHOR("training");
MODULE_DESCRIPTION("basic irq request and handler demo");
```

示例：

```dts

	compatible = "training,my-irq-demo";
```


不要随意占用真实硬件中断号，也不要随意映射未知寄存器地址。

## 19. 查看中断信息

Linux 中最常用的中断查看方式是：

```bash
cat /proc/interrupts
```

可以配合 `grep`：

```bash
cat /proc/interrupts | grep my_irq_demo
```

或者观察中断计数变化：

```bash
watch -n 1 'cat /proc/interrupts | grep my_irq_demo'
```

`/proc/interrupts` 中通常能看到：

```text
中断号
各 CPU 上触发次数
中断控制器信息
触发类型
中断名称
```

例如：

```text
45:  0  12  0  0  GICv3  77 Level  my_irq_demo
```

不同平台显示格式会不同，重点关注：

```text
中断计数是否增长
中断名称是否是驱动注册时的 devname
是否分布到预期 CPU
是否出现异常高频增长
```

## 20. dmesg调试

加载模块后查看：

```bash
dmesg | tail -n 50
```

实时观察：

```bash
dmesg -w
```

建议在 `probe()` 中打印：

```c
dev_info(dev, "irq = %d\n", priv->irq);
```

在中断处理函数中，入门实验可以少量打印：

```c
dev_info(priv->dev, "irq handled, status = 0x%08x\n", status);
```

但真实项目中不要在高频中断里大量打印日志，否则会严重影响系统性能，甚至导致系统卡顿。

更好的方式是：

1. 用计数器记录中断次数；
2. 通过 sysfs/debugfs 读取计数；
3. 只在异常时打印错误日志。

## 21. 常见错误1：DTS中没有interrupts

如果 DTS 中没有 `interrupts`，但驱动调用：

```c
platform_get_irq(pdev, 0);
```

通常会失败。

现象：

```text
probe 进入了
platform_get_irq() 返回负数
request_irq() 没有执行
```

排查：

```bash
find /proc/device-tree/ -name '*irq*'
hexdump -C /proc/device-tree/路径/interrupts
dmesg | tail -n 50
```

---

## 22. 常见错误2：interrupts写错

如果 `interrupts` 中断号或触发类型写错，可能出现：

1. `platform_get_irq()` 失败；
2. 中断申请成功但永远不触发；
3. 中断触发异常；
4. 中断风暴；
5. 系统卡死；
6. 触发了别的设备的中断。

排查时要对照：

```text
SoC 手册
中断控制器 binding 文档
已有 DTS 示例
原理图或硬件连接说明
```

不要凭感觉填写中断号。

---

## 23. 常见错误3：request_irq失败

`devm_request_irq()` 失败可能是：

```text
1. irq number 无效
2. 中断已经被其他驱动独占
3. irqflags 不匹配
4. handler 为空
5. dev_id 使用不当
```

代码中必须检查返回值：

```c
ret = devm_request_irq(dev, irq, my_irq_handler, 0,
		       dev_name(dev), priv);
if (ret) {
	dev_err(dev, "failed to request irq: %d\n", ret);
	return ret;
}
```

如果返回 `-EBUSY`，常见含义是中断资源已经被占用。

---

## 24. 常见错误4：中断处理函数没有清中断

如果硬件要求软件清中断，但 handler 中没有清除中断状态，可能造成中断风暴。

现象：

```text
1. /proc/interrupts 中计数快速增长
2. CPU 占用率很高
3. dmesg 刷屏
4. 系统变慢甚至卡死
```

处理方式：

```text
根据硬件手册，在 handler 中正确清除中断状态。
```

注意 W1C 位，不要乱做读改写。

---

## 25. 常见错误5：中断处理函数里睡眠

错误示例：

```c
static irqreturn_t my_irq_handler(int irq, void *data)
{
	msleep(10);
	return IRQ_HANDLED;
}
```

普通硬中断上下文中不能睡眠。

如果确实需要耗时处理，应使用 threaded IRQ 或 workqueue。入门阶段先把硬中断 handler 写短。

---

## 26. 常见错误6：共享中断不判断状态

如果申请了 `IRQF_SHARED`，但 handler 不判断本设备状态，直接返回 `IRQ_HANDLED`：

```c
static irqreturn_t my_irq_handler(int irq, void *data)
{
	return IRQ_HANDLED;
}
```

这会影响同一条共享中断线上的其他设备。

正确方式：

```c
status = readl(priv->base + MY_REG_IRQ_STATUS);
if (!status)
	return IRQ_NONE;

/* handle and clear */

return IRQ_HANDLED;
```

---

## 27. 常见错误7：高频中断中大量打印

中断处理函数中大量打印：

```c
dev_info(priv->dev, "interrupt\n");
```

如果中断频率很高，会导致：

1. 日志刷屏；
2. 系统变慢；
3. 丢失真正有价值的日志；
4. 影响实时性；
5. 甚至诱发新的问题。

建议：

```text
低频实验可以打印；
真实驱动中尽量不要在高频中断路径中打印普通日志；
异常日志也要限频。
```

---

## 28. 中断调试推荐顺序

中断不工作时，按下面顺序排查：

```text
1. probe 是否进入？
2. compatible 是否匹配？
3. reg 是否映射成功？
4. DTS 中是否有 interrupts？
5. platform_get_irq() 是否成功？
6. devm_request_irq() 是否成功？
7. /proc/interrupts 中是否出现设备名？
8. 硬件是否真的使能了中断？
9. 设备侧中断使能位是否打开？
10. 中断状态寄存器是否置位？
11. handler 是否被调用？
12. handler 是否正确清中断？
13. 触发类型是否正确？
14. 是否存在中断风暴？
```

常用命令：

```bash
dmesg -w
cat /proc/interrupts
watch -n 1 'cat /proc/interrupts'
find /proc/device-tree/ -name '*demo*'
hexdump -C /proc/device-tree/路径/interrupts
find /sys/bus/platform/devices/ -name '*demo*'
```

## 29. 实习生实验要求

建议实习生基于前面的 `platform driver` 和 `ioremap` 实验继续扩展。

## 29.1 实验1：添加interrupts资源

在 DTS 节点中添加：

```dts
interrupts = <0 45 4>;
```

要求：

1. 重新编译并部署 DTB；
2. 确认运行时设备树中存在 `interrupts`；
3. 记录 `hexdump -C` 输出。

## 29.2 实验2：获取irq

在 `probe()` 中添加：

```c
priv->irq = platform_get_irq(pdev, 0);
if (priv->irq < 0)
	return priv->irq;

dev_info(dev, "irq = %d\n", priv->irq);
```

要求：

1. `probe()` 正常执行；
2. `dmesg` 中打印 irq number；
3. 说明这个 irq number 是 Linux irq number，不一定等于硬件中断号。

## 29.3 实验3：申请中断

添加：

```c
ret = devm_request_irq(dev, priv->irq, my_irq_handler, 0,
		       dev_name(dev), priv);
if (ret)
	return ret;
```

要求：

1. `request_irq` 成功；
2. `/proc/interrupts` 中能看到设备名；
3. 记录对应行。

## 29.4 实验4：观察中断计数

触发硬件中断后观察：

```bash
watch -n 1 'cat /proc/interrupts | grep my_irq_demo'
```

要求：

1. 记录触发前后的中断计数；
2. 判断 handler 是否被调用；
3. 如果计数不增长，按排查顺序分析原因。

## 29.5 实验5：故意删除interrupts

临时删除 DTS 中的 `interrupts` 属性，观察：

1. `probe()` 是否进入；
2. `platform_get_irq()` 返回什么；
3. `dmesg` 中有什么错误；
4. 驱动是否绑定成功。

这个实验用于理解：

```text
compatible 负责匹配；
reg 负责寄存器资源；
interrupts 负责中断资源。
```

## 31. 小结

Linux 驱动中断处理的基本流程是：

```text
DTS interrupts 描述中断资源
  ↓
platform_get_irq() 获取 Linux irq number
  ↓
devm_request_irq() 注册中断处理函数
  ↓
硬件触发中断
  ↓
内核调用 handler
  ↓
handler 读取状态、处理事件、清除中断
  ↓
返回 IRQ_HANDLED
```

入门阶段要牢记：

```text
不要硬编码 irq number；
不要忽略 platform_get_irq() 返回值；
不要忽略 devm_request_irq() 返回值；
不要在普通中断处理函数中睡眠；
不要忘记清中断；
不要在高频中断中大量打印；
共享中断必须判断是否为本设备中断。
```

中断调试最重要的几个观察点是：

```text
dmesg
/proc/interrupts
设备状态寄存器
中断状态寄存器
中断使能寄存器
```

把中断申请和中断处理函数理解清楚后，后续学习 GPIO 中断、I2C/SPI 控制器中断、DMA 完成中断、网卡中断等内容都会更容易。
