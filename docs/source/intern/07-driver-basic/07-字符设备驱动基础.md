# 07-字符设备驱动基础

## 1. 文档目标

本文用于讲解 Linux 字符设备驱动的基础概念和最小开发流程。

学习完本文后，应能够理解：

1. 什么是字符设备；
2. `/dev/xxx` 设备节点的作用；
3. 主设备号和次设备号的含义；
4. `file_operations` 如何连接用户态和内核态；
5. `open/read/write/release` 函数的基本写法；
6. 如何编译、加载、测试一个最小字符设备驱动。

本文目标不是把字符设备驱动讲到非常深入，而是让实习生能够建立基本认识，并能跑通一个最小实验。

---

## 2. 字符设备是什么

Linux 中的设备大致可以分为几类：

| 类型 | 典型设备 | 特点 |
|---|---|---|
| 字符设备 | 串口、GPIO调试设备、简单传感器、自定义控制设备 | 按字节流或命令方式访问 |
| 块设备 | 硬盘、NVMe、SD卡 | 按块读写，通常带缓存和文件系统 |
| 网络设备 | 网卡 | 通过网络协议栈收发数据包 |

字符设备的典型特点是：

1. 用户空间通常通过 `/dev/xxx` 访问；
2. 访问方式类似普通文件；
3. 支持 `open()`、`read()`、`write()`、`close()` 等文件操作；
4. 驱动内部通过 `file_operations` 提供这些操作的实现。

例如，用户空间执行：

```bash
cat /dev/mychardev
echo hello > /dev/mychardev
```

本质上会进入驱动中的 `read()` 和 `write()` 回调函数。

---

## 3. `/dev/xxx` 设备节点的作用

在 Linux 中，用户空间通常不能直接调用内核驱动函数。

用户空间访问驱动，一般要通过一个入口。对于字符设备来说，这个入口通常就是：

```text
/dev/xxx
```

例如：

```text
/dev/ttyS0
/dev/null
/dev/random
/dev/input/event0
/dev/mychardev
```

这些设备节点看起来像文件，但它们背后对应的是内核中的设备驱动。

用户空间程序调用：

```c
open("/dev/mychardev", O_RDWR);
read(fd, buf, len);
write(fd, buf, len);
close(fd);
```

内核会根据设备节点对应的设备号，找到对应的字符设备驱动，并调用驱动注册的文件操作函数。

---

## 4. 主设备号和次设备号

Linux 使用设备号来区分设备。

设备号由两部分组成：

```text
设备号 = 主设备号 + 次设备号
```

| 名称 | 含义 |
|---|---|
| 主设备号 major | 表示由哪一类驱动处理 |
| 次设备号 minor | 表示同一驱动下的哪个具体设备 |

例如：

```bash
ls -l /dev/null
```

可能看到类似输出：

```text
crw-rw-rw- 1 root root 1, 3 /dev/null
```

其中：

```text
c     表示字符设备
1     表示主设备号
3     表示次设备号
```

在简单实验中，通常使用内核动态分配设备号：

```c
alloc_chrdev_region(&devno, 0, 1, "mychardev");
```

这样可以避免手工指定主设备号导致冲突。

## 5. 字符设备驱动的基本组成

一个最小字符设备驱动通常包含下面几部分：

```text
字符设备驱动
 ├── 设备号 dev_t
 ├── struct cdev
 ├── struct file_operations
 ├── open/read/write/release 回调函数
 ├── class
 └── device 节点
```

核心关系可以理解为：

```text
用户程序
  ↓ open/read/write
/dev/mychardev
  ↓ 设备号
struct cdev
  ↓
file_operations
  ↓
驱动函数
```

这里最重要的是 `file_operations`。

---

## 6. file_operations 的作用

`struct file_operations` 用来描述字符设备支持哪些文件操作。

常见写法如下：

```c
static const struct file_operations mychardev_fops = {
	.owner = THIS_MODULE,
	.open = mychardev_open,
	.read = mychardev_read,
	.write = mychardev_write,
	.release = mychardev_release,
};
```

其中：

| 成员 | 对应用户态操作 |
|---|---|
| `.open` | `open()` |
| `.read` | `read()` |
| `.write` | `write()` |
| `.release` | `close()` |
| `.unlocked_ioctl` | `ioctl()` |
| `.mmap` | `mmap()` |
| `.poll` | `poll()` / `select()` |

本文只讲最基础的：

```text
open / read / write / release
```

`ioctl`、`mmap`、`poll` 后续再单独学习。

## 7. 最小字符设备驱动代码

下面是一个最小可运行的字符设备驱动示例。

文件名：

```text
mychardev.c
```

代码：

```c
#include <linux/module.h>
#include <linux/init.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/device.h>
#include <linux/uaccess.h>

#define MYCHARDEV_NAME	"mychardev"
#define MYCHARDEV_CLASS	"mychardev_class"
#define MYCHARDEV_BUFSIZE	128

static dev_t mychardev_devno;
static struct cdev mychardev_cdev;
static struct class *mychardev_class;
static struct device *mychardev_device;

static char kernel_buf[MYCHARDEV_BUFSIZE] = "hello from kernel\n";

static int mychardev_open(struct inode *inode, struct file *file)
{
	pr_info("mychardev: open\n");
	return 0;
}

static ssize_t mychardev_read(struct file *file, char __user *buf,
				      size_t count, loff_t *ppos)
{
	size_t len;

	len = strlen(kernel_buf);

	if (*ppos >= len)
		return 0;

	if (count > len - *ppos)
		count = len - *ppos;

	if (copy_to_user(buf, kernel_buf + *ppos, count))
		return -EFAULT;

	*ppos += count;

	return count;
}

static ssize_t mychardev_write(struct file *file, const char __user *buf,
				       size_t count, loff_t *ppos)
{
	if (count >= MYCHARDEV_BUFSIZE)
		count = MYCHARDEV_BUFSIZE - 1;

	memset(kernel_buf, 0, sizeof(kernel_buf));

	if (copy_from_user(kernel_buf, buf, count))
		return -EFAULT;

	kernel_buf[count] = '\0';

	pr_info("mychardev: write: %s\n", kernel_buf);

	return count;
}

static int mychardev_release(struct inode *inode, struct file *file)
{
	pr_info("mychardev: release\n");
	return 0;
}

static const struct file_operations mychardev_fops = {
	.owner = THIS_MODULE,
	.open = mychardev_open,
	.read = mychardev_read,
	.write = mychardev_write,
	.release = mychardev_release,
};

static int __init mychardev_init(void)
{
	int ret;

	ret = alloc_chrdev_region(&mychardev_devno, 0, 1, MYCHARDEV_NAME);
	if (ret)
		return ret;

	cdev_init(&mychardev_cdev, &mychardev_fops);
	mychardev_cdev.owner = THIS_MODULE;

	ret = cdev_add(&mychardev_cdev, mychardev_devno, 1);
	if (ret)
		goto err_unregister_chrdev;

	mychardev_class = class_create(THIS_MODULE, MYCHARDEV_CLASS);
	if (IS_ERR(mychardev_class)) {
		ret = PTR_ERR(mychardev_class);
		goto err_cdev_del;
	}

	mychardev_device = device_create(mychardev_class, NULL,
					   mychardev_devno, NULL,
					   MYCHARDEV_NAME);
	if (IS_ERR(mychardev_device)) {
		ret = PTR_ERR(mychardev_device);
		goto err_class_destroy;
	}

	pr_info("mychardev: init ok, major=%d minor=%d\n",
		MAJOR(mychardev_devno), MINOR(mychardev_devno));

	return 0;

err_class_destroy:
	class_destroy(mychardev_class);
err_cdev_del:
	cdev_del(&mychardev_cdev);
err_unregister_chrdev:
	unregister_chrdev_region(mychardev_devno, 1);
	return ret;
}

static void __exit mychardev_exit(void)
{
	device_destroy(mychardev_class, mychardev_devno);
	class_destroy(mychardev_class);
	cdev_del(&mychardev_cdev);
	unregister_chrdev_region(mychardev_devno, 1);

	pr_info("mychardev: exit\n");
}

module_init(mychardev_init);
module_exit(mychardev_exit);

MODULE_LICENSE("GPL");
MODULE_AUTHOR("example");
MODULE_DESCRIPTION("simple character device driver");
```

## 8. Makefile

文件名：

```text
Makefile
```

内容：

```makefile
obj-m += mychardev.o

KDIR ?= /lib/modules/$(shell uname -r)/build
PWD := $(shell pwd)

all:
	$(MAKE) -C $(KDIR) M=$(PWD) modules

clean:
	$(MAKE) -C $(KDIR) M=$(PWD) clean
```

注意：

1. `obj-m += mychardev.o` 表示编译为内核模块；
2. `KDIR` 指向当前内核的构建目录；
3. `M=$(PWD)` 表示当前目录是外部模块目录；
4. Makefile 命令前必须是 Tab，不是空格。

## 13. 代码流程分析

## 13.1 模块加载流程

模块加载时执行：

```c
module_init(mychardev_init);
```

也就是执行：

```c
mychardev_init()
```

初始化函数中做了这些事情：

```text
alloc_chrdev_region()
  ↓
cdev_init()
  ↓
cdev_add()
  ↓
class_create()
  ↓
device_create()
  ↓
生成 /dev/mychardev
```

其中：

| 函数 | 作用 |
|---|---|
| `alloc_chrdev_region()` | 动态申请设备号 |
| `cdev_init()` | 初始化字符设备对象 |
| `cdev_add()` | 把字符设备注册到内核 |
| `class_create()` | 创建 sysfs class |
| `device_create()` | 创建设备对象，并配合 udev 生成 `/dev` 节点 |

---

## 13.2 用户读取流程

用户执行：

```bash
cat /dev/mychardev
```

大致流程是：

```text
cat
  ↓
open("/dev/mychardev")
  ↓
mychardev_open()
  ↓
read()
  ↓
mychardev_read()
  ↓
copy_to_user()
  ↓
用户空间得到数据
  ↓
close()
  ↓
mychardev_release()
```

重点是：

```c
copy_to_user(buf, kernel_buf + *ppos, count)
```

它负责把内核空间的数据复制到用户空间。

---

## 13.3 用户写入流程

用户执行：

```bash
echo "hello driver" > /dev/mychardev
```

大致流程是：

```text
echo
  ↓
open("/dev/mychardev")
  ↓
mychardev_open()
  ↓
write()
  ↓
mychardev_write()
  ↓
copy_from_user()
  ↓
数据进入内核 buffer
  ↓
close()
  ↓
mychardev_release()
```

重点是：

```c
copy_from_user(kernel_buf, buf, count)
```

它负责把用户空间的数据复制到内核空间。

## 14. 为什么不能直接访问用户指针

在驱动的 `read()` 和 `write()` 中，可以看到参数里有：

```c
char __user *buf
const char __user *buf
```

`__user` 表示这个指针来自用户空间。

内核代码不能直接这样访问：

```c
kernel_buf[0] = buf[0];
```

原因是：

1. 用户空间地址和内核空间地址不同；
2. 用户传入的地址可能非法；
3. 直接访问可能导致内核崩溃；
4. 需要通过专门的接口进行安全复制。

因此应该使用：

```c
copy_to_user()
copy_from_user()
```

这是字符设备驱动中非常重要的基础点。

## 15. 常见错误与排查

## 15.1 Makefile 报错

如果看到类似错误：

```text
No rule to make target
```

常见原因：

1. Makefile 缩进不是 Tab；
2. `KDIR` 路径不存在；
3. 没有安装当前内核对应的 headers 或 kernel-devel；
4. 当前系统没有完整的编译环境。

检查：

```bash
ls /lib/modules/$(uname -r)/build
```

如果该路径不存在，需要安装匹配当前内核的开发包。

---

## 15.2 insmod 失败

加载模块失败时，先执行：

```bash
dmesg | tail -n 30
```

常见原因：

| 现象 | 可能原因 |
|---|---|
| `Invalid module format` | 模块和当前运行内核版本不匹配 |
| `Operation not permitted` | 权限不足或安全策略限制 |
| `File exists` | 模块已经加载 |
| `Unknown symbol` | 依赖符号缺失或内核配置不支持 |

查看模块 vermagic：

```bash
modinfo mychardev.ko | grep vermagic
uname -r
```

如果版本不一致，说明模块不是为当前运行内核编译的。

---

## 15.3 没有生成 /dev/mychardev

如果模块加载成功，但没有 `/dev/mychardev`，可以检查：

```bash
dmesg | tail
ls /sys/class/mychardev_class/
```

可能原因：

1. `device_create()` 失败；
2. 系统没有运行 udev/mdev；
3. 权限或系统环境异常。

如果没有自动生成设备节点，也可以手工创建：

```bash
sudo mknod /dev/mychardev c <major> 0
sudo chmod 666 /dev/mychardev
```

其中 `<major>` 从 dmesg 打印中获取。

---

## 15.4 rmmod 失败

如果卸载时报错：

```text
Module mychardev is in use
```

可能是有进程正在打开设备节点。

可以检查：

```bash
sudo lsof /dev/mychardev
```

结束相关进程后再卸载：

```bash
sudo rmmod mychardev
```

## 17. 本文小结

字符设备驱动是 Linux 驱动开发入门中非常重要的一步。

它解决的是：

```text
用户空间程序如何访问内核驱动
```

核心主线是：

```text
/dev/xxx
  ↓
设备号
  ↓
struct cdev
  ↓
file_operations
  ↓
open/read/write/release
```

需要重点掌握：

1. 字符设备通过 `/dev/xxx` 暴露给用户空间；
2. 主设备号用于找到驱动，次设备号用于区分具体设备；
3. `file_operations` 是用户态文件操作和内核驱动函数之间的桥梁；
4. 用户空间和内核空间之间复制数据必须使用 `copy_to_user()` 和 `copy_from_user()`；
5. 字符设备实验能帮助初学者理解 Linux 驱动和用户程序之间的基本关系。
